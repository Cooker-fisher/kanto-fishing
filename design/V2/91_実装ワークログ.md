## T38-fix — build_fish_area_pages 孤児HTMLパージ（2026/05/16）

### 背景

T38 Phase A/B/C 完了後の検証で、`fish/index.html` と `area/index.html` の Phase C「全件展開セクション」が `265本中256本（96.6%）` しかカバーしていないことが判明。残り9本（`aji-katakai`・`bishi-aji-kanazawa-hakkei`・`hanadai/inada/kasago/tachiuo-hayama-abuzuri`・`ishidai-chigasaki`・`madai-fukuda`・`warasa-sajima`）は CSV 集計（`fish_area_summary`）に存在しないコンボだった。

### 根本原因

`build_fish_area_pages()` は当日 catches snapshot から `(fish, area)` ペアを構築して HTML を生成するが、過去のある日に当日 catches に紛れ込んだ単発レコードで HTML が生成されると、後日 normalize 結果から消えても **HTML だけ孤児として残る**仕様だった。

冒頭パージは `is_old_ogp` / `is_old_faq_v1` のみで、データ実体との整合チェックがなかった。

### 修正内容

`build_fish_area_pages()` 冒頭パージブロックを拡張（crawler.py +約75行）:

1. `_hist_rows_for_fa` を関数冒頭に前倒しロード（既存の二重ロード解消も兼ねる）
2. `_hist_fa_set` / `_fas_fa_set` / `_today_fa_set` の3つの (fish, area) セットを構築
   - `_hist_fa_set`: hist_rows 由来（is_cancellation 除外）
   - `_fas_fa_set`: `fish_area_summary` 引数由来
   - `_today_fa_set`: 当日 `data` から **T34 7日窓 cutoff** 適用済み（catches.json フォールバック対策）
3. `_valid_fa_set = _hist_fa_set | _fas_fa_set | _today_fa_set` を有効ペア集合とする
4. `_parse_fa_filename(stem)` ヘルパー: スラッグ→日本語の逆引き辞書で `(fish_jp, area_jp)` に分解。曖昧/未登録は `(None, None)` 返却で安全側にパージ対象外
5. パージブロックの判定に追加: 「既存条件（OGP/FAQv1）に該当しなければ、`(fj, aj)` を逆引きし、`_valid_fa_set` に存在しなければ孤児として `os.remove()`」
6. `_orphan_purge_count` ログ出力で追跡可能

### 検証結果

- `python crawl/validate_output.py` — errors=0 / warnings=0（全33不変条件 PASS）
