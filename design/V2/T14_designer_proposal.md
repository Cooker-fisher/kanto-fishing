## 2. 問題定義

index.html の「今日釣れている魚」ZONE B では、`_heat_score` が魚種を降順ソートして表示する。このスコアは `log1p(N) × ratio × season_mul` で計算されるため、`log1p` で対数圧縮されていても前週比 `ratio` が最大3倍まで乗算される。結果として「先週0→今週3船宿のカンパチ（ratio=3.0相当）」が「先週52→今週26件のワラサ（ratio=0.47）」を抜いてしまう。ユーザーが「N=4のカンパチがN=26のワラサ・N=45のシロギスより上位」と指摘した事象がこれに該当する。少数件の偶発釣果が前週比ブーストで主力魚を押しのける挙動は、ユーザーの意思決定支援という設計目的に反する。

---

## 3. 採用方針

**A案: N<5 の魚種を score × 0.01 で末尾に集める**

- 閾値: **N = 5**（件数ではなくレコード数 `len(cs)` が5未満）
- 押下げ係数: **0.01**（スコア 0.0 にしないことで同 N<5 グループ内での相対順序は保持）
- 根拠（researcher 調査より）:
  - 閾値3では N=4 のカンパチが依然上位に残り、本問題を解決しない
  - 閾値7以上では TOP12 に 4件以上の入れ替えが発生し副作用が大きすぎる
  - 閾値5は「単独船宿の偶発釣果（1〜4件）」と「複数船宿で継続する釣果（5件〜）」の自然な境界
  - 5月時点で N<5 かつ peak（旬係数=1.3）の魚種は存在しない → 旬魚の誤抑制は発生しない
  - 閾値5 では TOP12 の入れ替えは 0件（シミュレーション確認済み）

T12 R4 の決定内容は「ZONE Bカードの序列化はやらない（情報量は多い方が良い）」であり、具体的には「件数順・トレンド順などの明示的な並び替えUIや、絞り込みフィルターを追加しない」という意味だった。

## 5. 実装仕様（programmer 向け）

### 修正対象

`crawler.py` L5984-5999 `_heat_score` 関数のみ。

### 変更後の関数全文（コピペ可能）

```python
def _heat_score(fish, cs):
    # 件数を対数変換（絶対数の支配力を圧縮）
    cnt = math.log1p(len(cs))
    # 前週比（上限3倍）
    tw = get_yoy_data(history, fish, year, week_num)[0]
    pw = get_prev_week_data(history, fish, year, week_num)
    ratio = 1.0
    if tw and pw:
        ts = tw.get("ships") or 0
        ps = pw.get("ships") or 0
        if ts and ps:
            ratio = min(ts / ps, 3.0)
    # 旬係数
    sk, _ = _fish_signal(fish, current_month)
    season_mul = {"peak": 1.3, "season": 1.1, "normal": 1.0, "late": 0.7}[sk]
    score = cnt * ratio * season_mul
    # N<5 の魚種は末尾に集める（前週比ブーストで少数派が主力魚を抜くバグの修正）
    if len(cs) < 5:
        return score * 0.01
    return score
```

変更差分は末尾4行（`score = cnt * ratio * season_mul` への変数分離 + `if len(cs) < 5` ブロック）のみ。既存ロジックは一切変更しない。

### 副作用チェック

- TOP12 が崩壊しない: researcher シミュレーション（`4.1 閾値=5 の TOP12` 表）で入れ替え 0件を確認済み。現状1〜12位の全魚種が N≥5 のため、`score * 0.01` は適用されない。
- 不変条件 #1（魚種カード ≥ 5）: 今回の日付（2026/05/07 sparse, 39魚種認識）では N<5 の押下げ対象が17魚種。残り22魚種が N≥5 で通常スコア。TOP12 は全て N≥5 のため最低12カード以上が出力される。条件 ≥ 5 を大きく超える。
- L6001 ソート呼び出し: `sorted(fish_summary.items(), key=lambda x: -_heat_score(x[0], x[1]))` は変更なし。関数シグネチャは同一なので呼び出し側の修正は不要。
