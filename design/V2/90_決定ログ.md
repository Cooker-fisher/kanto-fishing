## 2026-05-24: chowari-NNNNN slug 114件を意味ある英語slug に刷新

### 背景

ユーザーが Google Search Console で `https://funatsuri-yoso.com/ship/chowari-00454.html` のような暫定 ID slug が index されているのを発見。「chowari-xxx という URL は絶対にダメ」「即刻変えること」と指示。

### 根本原因

`direct-crawl/register_other_chowari.py:52` が `slug = f"chowari-{cid}"` という暫定 slug を生成し、ships.json に登録した経緯。crawler.py の `build_ship_pages()` は `ship["romaji_slug"]` をそのままファイル名 / canonical / sitemap に流すため、SEO 上 NG な URL が公開された（114隻 / `docs/ship/chowari-*.html`）。

### 対応

1. **新 slug 命名規則**（既存パターン踏襲）:
   - 日本語名 → 音読みベース romaji
   - 「丸」「号」「釣船店」「遊船」「屋」「本家」等の屋号接尾辞は分離
   - 「第N」prefix は `daisan-`, `daini-` 等
   - 同名衝突は港 suffix（`yusei-maru-katase` / `yuko-maru-matsuwa` / `koei-maru-yaizu` / `hayabusa-maru-shiba` / `suzuki-maru-ajiro`）
   - 例: 大松丸 → `omatsu-maru.html` / 直重丸 → `naoshige-maru.html` / 健清丸 → `kensei-maru.html`

2. **実装ファイル**:
   - `crawl/rename_chowari_slugs.py`（114件の手動辞書 + ships.json 一括書き換え）
   - `crawl/chowari_slug_redirect_map.json`（旧→新 マッピング・redirect 用）
   - `crawler.py` `build_ship_redirects()` 追加: 旧 `chowari-NNNNN.html` を meta refresh + canonical=新URL + noindex の redirect HTML として regenerate
   - `crawl/replace_chowari_links.py`: docs/**/*.html 内の旧 slug 参照を新 slug に一括置換（413ファイル / 575リンク）

3. **Google index 済 URL の扱い**:
   - 旧 `chowari-*.html` ファイルは **削除せず redirect HTML として残す**（meta refresh 0秒 + canonical=新URL + `<meta name="robots" content="noindex,follow">`）
   - sitemap.xml には新 slug のみ列挙（旧 chowari-* は含めない）
   - これにより Google は徐々に新 URL に統合し、旧 URL は index から落ちる想定

### 結果

- docs/ship/ 内訳: 計 371 件（新slug 258 + redirect 114）
- 内部リンク 575件を新slugに置換（index・calendar・fish・area・fish_area・ship）
- sitemap.xml の chowari- 残存ゼロ
- `python crawl/validate_output.py` errors=0 / warnings=0

### 2026/05/22 — 海況コメントの風速表現を船釣り基準4段階に再設計

- **決定**:
  - 風速・波高に **内海/外海別の severity (sev 0=穏 / 1=やや / 2=強 / 3=暴) を導入**
  - 内海ベルト: 風 <6 / 6–8 / 8–10 / ≥10 m/s（東京湾・相模湾・湾内の小型船基準）
  - 外海ベルト: 風 <8 / 8–10 / 10–13 / ≥13 m/s（外房・銭洲・神津島の大型船基準）

### 2026/04/20 — ポイント抽出ルール（_split_point_places_depth 仕様確定）

- **決定**: `point_raw` 文字列から地名と水深を以下のルールで分離し CSV に記録する

  | 入力例 | point_place1 | point_place2 | depth_min | depth_max |
  |--------|-------------|-------------|-----------|-----------|
  | `走水沖水深50m` | 走水沖 | — | 50 | 50 |
  | `走水沖水深30m前後` | 走水沖 | — | 30 | 30 |
  | `金谷沖水深25〜40m` | 金谷沖 | — | 25 | 40 |
  | `観音崎沖～猿島沖` | 観音崎沖 | 猿島沖 | — | — |
  | `観音崎沖→猿島沖・水深60m` | 観音崎沖 | 猿島沖 | 60 | 60 |

  - 水深表現（`水深Xm`・`タナXm`・`Xm前後`・`X〜Ym`）は depth_min/depth_max 列に分離して記録し、地名部分から除去する
  - `前後`・`付近`・`他` は地名から除去する
  - 数値のみの文字列（`25~40m` など）は地名としない
  - 区切り文字: `・`（中黒）`→`（矢印）`/`（スラッシュ）`〜`（波ダッシュ）

- **理由**: 水深がポイント名に混入すると同一ポイントが複数IDになる（「走水沖」と「走水沖水深50m」が別ポイントとして集計される）。地名と水深は意味が異なる情報なので分離して管理する
- **実装**: `crawler.py` の `_split_point_places_depth()`
- **関連ロール**: プログラマー、分析者

### 2026/04/20 — ポイント別分析の設計方針

- **決定**: ポイントを分析単位とする場合は以下のルールに従う

  1. **複合ポイントは point_place1 を分析の単位とする**（例: `観音崎沖～猿島沖` → `観音崎沖` で分析）
  2. **N>=30 のポイントのみモデルを持つ**（MIN_N_COMBO=30 と同じ閾値）
  3. **ポイント別モデルは combo（船宿×魚種）モデルより優先して適用する**
  4. **点予測（どのポイントに行くか）は `combo_point_events` の月別集計から最頻ポイントを使う**
  5. **ポイント固有デカダルが旬6未満の場合は全体デカダルで補完する**

- **背景**: 複数ポイントを混ぜた1つのモデルでは季節バイアスが発生し精度が悪化する。例: タチウオ×吉野屋は夏=走水沖・冬=観音崎沖と季節でポイントが変わるため、混合モデルではポイント固有の気象相関が正しく取れない
- **DB テーブル**:
  - `combo_point_backtest`: ポイント別バックテスト精度（H=0,1,3,7,14,21,28）
  - `combo_point_wx_params`: ポイント別気象因子・係数
  - `combo_point_stats`: ポイント別件数・平均釣果
  - `combo_point_events`: ポイント選択の行レベルイベント（気象条件付き）
- **関連ロール**: 分析者

## 2026-05-01: データ消失 regression 防止策（3層 gatekeeper 導入）

### 背景

このセッションを含めて以下の data 消失 regression が複数回発生:

| 日 | 症状 | 根本原因 |
|---|---|---|
| 過去複数回 | fish/index.html が 1〜2 種だけ・area/index.html が 2 エリアだけ | `--html-only` モードが catches.json (今日 sparse) で全 HTML を再生成し rich data を上書き |
| 4/25〜4/30 | data/V2/2026-04.csv が 4/24 で停止 | GitHub Actions の `crawler.py --export-csv` が stale な catches_raw.json (4/22 まで) から CSV 全再生成し、save_daily_csv() の追記分を毎日 wipe |

### 防衛策（3層）

**Layer 1: コード側の冪等性**
- `--html-only` モード: index/fish/area/ship/forecast/sitemap を生成しない（CSS/JS/calendar のみ）
- `save_daily_csv()`: append-only + (ship, area, date, fish_raw) で dedup → 既存記録を消さない
- `--export-csv`: catches_raw.json の最新日付が today-7日 より古ければ ABORT（`FORCE_EXPORT=1` で override 可）

**Layer 2: CI の gatekeeper**
- `crawl/validate_output.py` を crawl.yml に組込
  - 6 つの不変条件をチェック（fish_index 釣果あり魚種 ≥ 5 / area_index ≥ 5 エリア / 当月CSV最新日付 today-2日以内 / catches_raw.json ≥ 50,000件 等）
  - 違反時は非 0 終了 → push スキップ → 壊れた状態がデプロイされない

**Layer 3: ワークフロー削減**
- crawl.yml から `--export-csv` ステップを削除（save_daily_csv が単独で CSV を維持）
- 直接クロール（shojiro/koueimaru）の point_raw 補完は手動で `FORCE_EXPORT=1 python crawler.py --export-csv` する運用に変更

### 不変条件（変えてはいけない契約）

1. `data/V2/*.csv` は append-only。手動再生成時は `FORCE_EXPORT=1` を明示
2. `catches.json` (data field) はスナップショット (today-only)。データ依存ページの生成元として使ってはならない
3. `_load_recent_catches_for_index()` は valid_catches と完全互換（count_range/size_cm/weight_kg を含む）の record を返す
4. ホームページの ZONE B (魚種カード) と ZONE B2 (エリア今日) は当日 sparse 時 (< 30 件) は 7日窓フォールバック。HERO/LIVE は当日セマンティクス維持
5. CI に `validate_output.py` を組み込んだまま運用する。gatekeeper を回避する変更は禁止

### 主要コミット

### 2026/05/08 — 表示仕様の確定（分析側からの共有・補遺3）

分析チーム（analysis/V2）で「商品的中率」KPI 議論の結果、出力表示仕様が確定。design/V2 側でも同方針に従う必要がある。

#### 確定事項（表示仕様）

| 要素 | 表示形式 |
|---|---|
| 数（匹）| `min匹 〜 max匹` |
| 型（cm）| `min cm 〜 max cm` |
| 重さ（kg）| `min kg 〜 max kg` |

#### 理由

ユーザー指摘:「匹数についてAveは不要。なぜならAveじゃないから。1,1,1,100 も 1,99,99,99,100 もどちらも 1〜100 になるため」

釣果報告における min/max は「同船者の幅」であり、avg は単純平均でしかなく統計的意味を持たない。釣り人にとっては「min がどこまで保証されるか」「max がどれくらい釣れる可能性があるか」が知りたい情報であり、avg を見ても意思決定に役立たない。

#### 4 つの禁止事項（補遺2 + 補遺3 統合）

1. cnt_min/max にそれぞれ予測区間（confidence interval）を被せた `[lo_of_lo, hi_of_lo]` 形式で出力する
2. レンジを階層化（meta-interval）する
3. min と max を別々のレンジ・別カードで表示する
4. **avg/ave/平均 を表示する**（補遺3 で追加）

### 2026-05-21 — 「/ と /index.html がズレる」現象への対策（不変条件 34 追加）

#### 観測現象

トップページの2つの URL `https://funatsuri-yoso.com/` と `https://funatsuri-yoso.com/index.html` が同じ瞬間に異なるバージョン（前者 5/20(水)・74件 / 後者 5/19(火)・65件）を返す瞬間があった。前回（同じ症状で逆向き）の修正後にまた発生。

#### 調査結果

`curl -sI` で両URLのレスポンスヘッダを比較:

| ヘッダ | / | /index.html |
|---|---|---|
| ETag | `"6a0e35ee-1398e"` | `"6a0e35ee-1398e"`（同一） |
| Last-Modified | `Wed, 20 May 2026 22:30:06 GMT` | `Wed, 20 May 2026 22:30:06 GMT`（同一） |
| Content-Length | 80270 | 80270（同一） |

→ **サーバー側にあるファイルは完全同一**。`crawler.py` は `docs/index.html` を1ファイルだけ生成しており、別経路で `/index.html` を生成していない。canonical は両方とも `https://funatsuri-yoso.com/`（trailing slash）に統一済み（`crawler.py:7498`）。

#### 原因

GitHub Pages の Fastly CDN は `/` と `/index.html` を **別キャッシュキー** で管理する。`Cache-Control: max-age=600`（10分・GitHub 側固定で変更不可）のため、片方のキャッシュが先に invalidate されると **最大10分間** ズレが見える。

#### 決定

1. **内部リンクから `index.html` 末尾の URL を完全排除**（全レベル徹底）:
   - ルート: `/index.html` → `/`、`../index.html` → `../`、`./index.html` → `./`、`index.html` → `/`
   - サブディレクトリ: `/forecast/index.html` → `/forecast/`、`/x_post/index.html` → `/x_post/`、その他 `/<subdir>/index.html` → `/<subdir>/`
   - `crawler.py` 29箇所（既存16 + レビュー後追加13）+ `docs/` 配下 HTML 全件機械置換
2. **副次バグ修正**: `premium/plan.html` パンくず「有料プラン」リンクが誤って `../index.html`（トップと同じ場所）を指していたのを `../forecast/` に修正（`crawler.py:10604` 第2リンク）
3. **不変条件 34 追加**: 全 `docs/**/*.html` で **`href` の値が `index.html` で終わる**内部リンクが0件であることを `validate_output.py` で検証（正規表現 `href=["'][^"']*index\.html["']` 使用）
4. mockup 系（`design/V2/mockup-*.html`）も追って統一（別タスク・docs 配下ではないので不変条件対象外）

#### 残存リスク

外部ブクマ・古いSNSシェア・直URL貼り付けで `/index.html` に来た人は引き続き CDN TTL 10分の影響を受ける。GitHub Pages 側の仕様で対策不可。canonical で SEO 上は `/` に統一済みなので、検索流入は時間と共に `/` に寄る。

#### レビュー後修正（スコープ拡張）

初回実装後の reviewer / persona-checker レビューで、当初スコープ「ルート `/` と `/index.html` のみ」では検出パターンと対象 URL が不完全と判明:

| 漏れ | 検出パターン |
|---|---|
| `calendar.html` パンくず `href="index.html"`（相対・スラッシュなし） | 当初パターン `href="/index.html"` から漏れ |
| `x_post/2026-05-XX.html` 21ファイル の「全日程一覧へ」`href="./index.html"` | 当初パターン から漏れ・同型 CDN ズレ問題 |

### 2026-05-22 — docs/index.html 「最新表示日が 5/19(火)」regression（手動 commit による巻き戻り）

#### 観測現象

5/22 00:13 JST 時点で `https://funatsuri-yoso.com/` を開くと、以下の表示が残っていた:

- `最終更新: 2026/05/21 19:12 JST`
- HERO: `5/19(火)の釣果報告 — 39船宿・23エリア`
- ZONE B 見出し: `5/19(火) 釣れている魚`
- LIVE ティッカー: 全行が `5/19(火) ... × ... XX匹` 形式

ユーザー指摘「もう21日の釣果がでてるでしょ？なんで19日が最新？」

#### 調査結果

`catches_raw.json` には `2026/05/21` のレコードが正常に格納済み。
`docs/index.html` の git 履歴を確認すると、5/21 のコミットが3つ:

| commit | 時刻 (JST) | 作者 | docs/index.html の表示日 |
|---|---|---|---|
| `fa66a2687` | 5/21 19:14 | ローカル手動 | 5/19(火)・39船宿・23エリア |
| `158090c85` | 5/21 20:01 | GitHub Actions | **5/21(木)・10船宿・9エリア** ✅ |
| `927a35bb5` | 5/21 23:11 | ローカル手動 | **5/19(火)・39船宿・23エリア** ❌ |

`git diff 158090c85 927a35bb5 -- docs/index.html` の結果:

```
- <span>5/21(木)の釣果報告 — 10船宿・9エリア</span>
+ <span>5/19(火)の釣果報告 — 39船宿・23エリア</span>
- <div class="updated">最終更新: 2026/05/21 19:52 JST</div>
+ <div class="updated">最終更新: 2026/05/21 19:12 JST</div>
- (LIVE ティッカー 全行 5/21(木) × イサキ × つる丸 50匹 等)
+ (LIVE ティッカー 全行 5/19(火) × ... 等)
```

→ **GitHub Actions が正しく 5/21(木) 表示に更新した docs/index.html を、その後の手動 commit が古い 5/19 版に上書きで戻した**。

#### 原因

`927a35bb5 fix(crawler): / と /index.html CDN キャッシュズレ対策` のコミット作業中、`/index.html` を `/` に置換するため `crawler.py` をローカル再実行 → `docs/index.html` 再生成 → commit。このとき:

1. ローカルの `catches.json` は5/21 20:01 の GitHub Actions push よりも前にローカルクロールした古い snapshot（5/19 ベース・5/21 のレコード未含有）
2. `_resolve_display_dataset(catches, "2026/05/21")` は 5/21 のレコードが0件のため `max(dates) = 2026/05/19` にフォールバック
3. 結果として「5/19(火)」表示の `docs/index.html` が生成された
4. それを commit & push → GitHub Actions の正しい 5/21 表示版を上書き

つまり **GitHub Actions のクロール結果を `git pull` せずに `crawler.py` をローカル再実行すると、古い catches.json で HTML 生成 → 表示日が巻き戻る**。

#### 決定

**ローカルで `crawler.py` を再実行する前に必ず `git pull` する**。これを開発手順として固定。

具体的なケース分岐:

| 作業目的 | 手順 |
|---|---|
| HTML の文字列置換だけしたい（リンク変換等） | `crawler.py` を実行せず、`sed`/Python script で `docs/` を直接置換 |
| `crawler.py` のロジック変更を反映したい | 1. `git pull`（最新 catches.json を取得） 2. `python crawler.py` 実行 3. validate_output 実行 4. commit & push |
| クロール結果だけ最新化したい | GitHub Actions の workflow_dispatch を使う（ローカル `crawler.py` を回さない） |

不変条件には**追加しない**。理由:

- 不変条件 34（`href` から `index.html` 排除）と異なり、これは「HTML 生成元データの鮮度」の問題で、`docs/` 自体に静的な不変パターンが存在しない（日付ラベルは毎日変わる）
- 「`docs/index.html` の表示日が catches_raw の最新日 - N 日以内」のような閾値化は、当日 sparse 時のフォールバック挙動と競合する
- 代わりに `REGRESSION_PREVENTION.md` の「設計契約」セクションに **ローカル `crawler.py` 実行前 `git pull` 必須** を明文化する

#### 復旧

5/22 16:30 JST の GitHub Actions 自動実行で自動修復見込み（5/22 の最新クロール結果で `docs/index.html` 再生成）。緊急復旧が必要なら workflow_dispatch 手動キックでも可。
