### 2026/04/12 — combo_deep_dive 並列実行を標準とする

- **決定**: 全魚種 deep dive 実行は `run_full_deepdive.py --workers 4` を使う。逐次実行は禁止
- **理由**: 42〜55魚種を逐次実行すると30〜40分かかる。workers=4 並列で約1/4に短縮（SQLite WAL モードにより同時書き込みは安全にシリアライズ）
- **実行コマンド**:
  ```bash
  # 標準（全魚種 workers=4）
  python analysis/V2/methods/run_full_deepdive.py --workers 4

  # 特定魚種のみ再実行
  python analysis/V2/methods/run_full_deepdive.py アジ マダイ --workers 2

  # GitHub Actions（2CPU環境）
  python analysis/V2/methods/run_full_deepdive.py --workers 2
  ```
- **禁止パターン**: `for fish in fish_list: subprocess.run(...)` のような手書きループ。run_full_deepdive.py を経由すること
- **workers 推奨値**: ローカルSSD=4〜6 / GitHub Actions(2CPU)=2〜3 / HDD=2

### 2026/04/20 — ポイント抽出ルール（_split_point_places_depth 仕様確定）

- **決定**: `point_raw` 文字列から地名と水深を以下のルールで分離し CSV に記録する

  | 入力例 | point_place1 | point_place2 | depth_min | depth_max |
  |--------|-------------|-------------|-----------|-----------|
  | `走水沖水深50m` | 走水沖 | — | 50 | 50 |
  | `走水沖水深30m前後` | 走水沖 | — | 30 | 30 |
  | `金谷沖水深25〜40m` | 金谷沖 | — | 25 | 40 |
  | `観音崎沖～猿島沖` | 観音崎沖 | 猿島沖 | — | — |
  | `観音崎沖→猿島沖・水深60m` | 観音崎沖 | 猿島沖 | 60 | 60 |

  - 水深表現（`水深Xm`・`タナXm`・`Xm前後`・`X〜Ym`）は depth_min/depth_max 列に分離して記録し、地名部分から除去する
  - `前後`・`付近`・`他` は地名から除去する
  - 数値のみの文字列（`25~40m` など）は地名としない
  - 区切り文字: `・`（中黒）`→`（矢印）`/`（スラッシュ）`〜`（波ダッシュ）

- **理由**: 水深がポイント名に混入すると同一ポイントが複数IDになる（「走水沖」と「走水沖水深50m」が別ポイントとして集計される）。地名と水深は意味が異なる情報なので分離して管理する
- **実装**: `crawler.py` の `_split_point_places_depth()`
- **関連ロール**: プログラマー、分析者

### 2026/04/20 — ポイント別分析の設計方針

- **決定**: ポイントを分析単位とする場合は以下のルールに従う

  1. **複合ポイントは point_place1 を分析の単位とする**（例: `観音崎沖～猿島沖` → `観音崎沖` で分析）
  2. **N>=30 のポイントのみモデルを持つ**（MIN_N_COMBO=30 と同じ閾値）
  3. **ポイント別モデルは combo（船宿×魚種）モデルより優先して適用する**
  4. **点予測（どのポイントに行くか）は `combo_point_events` の月別集計から最頻ポイントを使う**
  5. **ポイント固有デカダルが旬6未満の場合は全体デカダルで補完する**

- **背景**: 複数ポイントを混ぜた1つのモデルでは季節バイアスが発生し精度が悪化する。例: タチウオ×吉野屋は夏=走水沖・冬=観音崎沖と季節でポイントが変わるため、混合モデルではポイント固有の気象相関が正しく取れない
- **DB テーブル**:
  - `combo_point_backtest`: ポイント別バックテスト精度（H=0,1,3,7,14,21,28）
  - `combo_point_wx_params`: ポイント別気象因子・係数
  - `combo_point_stats`: ポイント別件数・平均釣果
  - `combo_point_events`: ポイント選択の行レベルイベント（気象条件付き）
- **関連ロール**: 分析者

### 2026/04/26 — HEROゾーン日付ラベル明示化（T9）

#### 決定

- **「本日の釣果報告 — XX船宿・XXエリア」のラベルを実データ日付に自動切替**する
  - 当日データあり: `今日の釣果報告 — XX船宿・XXエリア`
  - 当日0件（フォールバック時）: `4/25(土)の釣果報告 — XX船宿・XXエリア`
- **「最終クロール」→「最終更新」に語を変更**し、JST明記を維持（`最終更新: 2026/04/26 10:52 JST`）
- **`datetime.now()` を `datetime.now(JST).replace(tzinfo=None)` に全置換**（25箇所）
  - 理由: GitHub Actions コンテナがUTCのため、cron 16:30 JST 実行でも crawled_at が UTC 表記になっていた
- **対象ページ**: index.html / fish/*.html / area/*.html / ship/*.html（4種・100ファイル超）

#### 背景・きっかけ

- ユーザーが「本日の釣果報告 — 24船宿・18エリア」を見て「70船宿のうち1/3しか釣り行ってない？」と疑問
- 調査結果: 24船宿は実は前日(4/25土)のデータ。当日(4/26)0件のフォールバックを「本日」と表示していた
- 4/25土曜の24船宿は金曜・土曜の中央値レベル（決して少なくない）

#### 実装内容

- **新ヘルパー関数**:
  - `_format_date_label(date_str)`: `YYYY/MM/DD` → `M/D(曜)` 変換（例: `4/25(土)`）
  - `_resolve_display_dataset(catches, today_str)`: 当日データありなら（catches, '今日', today_str）、なければ最新日にフォールバック
- **修正箇所（最終的に9箇所）**:
  - index HEROラベル / 最終更新表記
  - fish/*.html `.fh-m` / 「今日の釣果」見出し / 7日バーグラフ最右列 / fish-cardミニバーグラフ最右列 / area_cmp_html h3
  - area/*.html `.ah-m` / title / OGP / meta description（「【今日N件】」を `fish_label` に置換）
  - ship/*.html 7日バーグラフ最右列

#### スコープから除外（責任者判断）

- index.html ZONE B 見出し「今日 釣れている魚」「エリア別 今日の釣果」「今日ほかに釣れている魚」
  - 理由: HEROで日付明示済み。セクション見出しまで全文置換すると日常語が消えて違和感が出る
- chigasaki / edogawa / heiwajima 等、catches.json に当日データがないエリアの古いHTML
  - 理由: 既存仕様で再生成対象外。翌日以降データが入れば自動で更新される
