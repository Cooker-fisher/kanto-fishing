# 釣果価値チェッカー — データ

このドキュメントは Phase 1 で使うすべてのデータ設計を定義する。

- 卸値クロール体制（東京都中央卸売市場 月報 明細データ）
- 魚種対比表（JSONに移管 → `fish-species-map.json`）
- 倍率テーブル（卸値 → 小売値）
- サイズ・重量による単価変動

魚種ごとの個別マッピングは **`fish-species-map.json`** が SoT（唯一の真実）。本MDは方針・運用ルールを定義する。

### 1.3 価格計算式

```
平均卸値 (円/kg) = sum(金額) / sum(数量)
```

魚種ごとに「市場・出荷地」を全集計して単一の平均卸値を出す。

レンジ表示用に、出荷地別の最高・最低も持つ:

```
卸値 high (円/kg) = max(出荷地別の 金額/数量)
卸値 low  (円/kg) = min(出荷地別の 金額/数量)
```

### 1.4 フィルタ条件

| 列 | 採用 | 除外 |
|---|---|---|
| 大分類 | 「鮮　魚」「活魚類」のみ | 冷凍魚・加工品・貝類・海そう類・淡水魚 |
| 数量(kg) | > 0 | 0 や NULL は除外 |
| 出荷地 | 全件採用（国産・輸入区別なし）| ― |

### 1.5 天然優先ルール（重要）

**釣果は基本天然**。月報の同一魚種で複数候補がある場合、以下の優先順で採用する:

```
天然（鮮魚）  ＞  活魚（活魚類）  ＞  養殖（鮮魚・活魚）
```

例:

| 月報品目 | 採用 | 備考 |
|---|---|---|
| またい（天然）854円/kg | ✓ 採用 | 釣果はこれで換算 |
| またい（養殖）1,303円/kg | 参考のみ | レンジ上限の目安に使う可 |
| 活またい 1,292円/kg | 副 | 同一 price_fish_id で別レンジ持つ |
| かんぱち（天然）1,552円/kg | ✓ 採用 | |
| かんぱち（養殖）2,621円/kg | 参考 | 釣果が稀に大型のときレンジ上限可 |
| 活かんぱち 2,391円/kg | 副 | |


```jsonc
{

      "prices": [
        {
          "price_fish_id": "maaji",
          "geppo_item": "まあじ",
          "qty_kg": 979941,
          "amount_yen": 690728205,
          "avg_yen_per_kg": 705,
          "high_yen_per_kg": 1850,
          "low_yen_per_kg":   420,
          "shipper_count": 23
        }
      ]
    }
  ]
}
```

**月1で `months` 配列に追記**。過去データは履歴保持のみ。

**表示用代表値は最新月単独**（`months[-1]`）を採用する（2026/05/24 仕様変更）。

理由:
- 魚種価格は **季節性が極めて大きい**（旬月と非旬月で 2〜3倍違う）
- 数年スパンでは **インフレ影響**（年率 2-5% 程度）が累積し古い数値は実態から乖離
- 直近12か月平均は「旬月の高値と非旬月の安値を平均化」してしまい、現在この瞬間の取引価値からズレる
- 釣果価値チェッカーは「**今の価値**」を出すツールであり、年平均ではなく直近値が正解

最新月以外の過去データは:
- 将来の検証用（後付け精度分析）
- 季節性可視化（将来 Phase で表示するなら）

として保持するが、`fish-price-master.json` 生成時には参照しない。

### 1.7 クロールスクリプト（実装済み）

**配置**: `docs/fish-value/crawl_wholesale.py`

**依存**: 標準ライブラリ + `openpyxl`（xlsx読み取り）

**使い方**:

```bash
# 初回バックフィル（202502〜直近全月）
python docs/fish-value/crawl_wholesale.py --backfill

# 月次運用（未取得月のみ追加）
python docs/fish-value/crawl_wholesale.py

# 特定月のみ
python docs/fish-value/crawl_wholesale.py --month 202604

# ローカル xlsx を直接読込（オフライン投入）
python docs/fish-value/crawl_wholesale.py --file path/to/s202604meisai_*.xlsx

# 既存月の上書き
python docs/fish-value/crawl_wholesale.py --force --month 202604
```

**動作（2026/05/20 設計変更）**:

1. `urls_manifest.json` を読み込み（yyyymm → 直URL の固定マッピング）
2. 各月の xlsx を Chrome UA で DL（0.8秒wait・既存 crawler.py 方針に揃える）
3. 大分類「鮮魚」「活魚類」のみ採用（冷凍/加工/貝/海そう/淡水は除外）
4. 品目別に集計（数量・金額・avg/high/low円/kg・出荷地数）
5. `wholesale-prices.json` の `months[]` に upsert

**月報ページHTMLからのリンク自動抽出は廃止**。理由:

- Actions runner からアクセスすると Cookie 同意ページ等の異なる HTML が返り、リンク検出0件で失敗するため
- ブラウザでは取得できるため User-Agent / Cookie 等の差異が原因と推定
- 確実性を優先して URL を固定マニフェスト管理に切替

**月次運用のマニフェスト更新（手動）**:

新月分は毎月20日頃に東京都市場の月報ページに追加公開される。`urls_manifest.json` の `months[]` に1行追記する:

```jsonc
{"yyyymm": "202605", "url": "https://www.shijou.metro.tokyo.lg.jp/documents/d/shijou/s202605meisai_YYYYMMDD"}
```

公開日（URL末尾のYYYYMMDD）はブラウザで月報ページを開いて該当行をコピペして確認する。追記したら PR or 直接 main に commit。21日の自動cronで取得される。

**仕様メモ**:

- `price_fish_id` 列は **「最初にマッチしたpfid」** が入る（同一 `geppo_item` を複数 pfid が共有する場合・例: アジ/ビシアジ → maaji）。アプリ側参照は `fish-species-map.json` の `geppo_item` から逆引きする方が安全
- マッピング外の品目も `price_fish_id: null` で保持（将来の魚種追加時に再利用）
- 同一 `yyyymm` が既に存在すれば skip（`--force` で上書き）

**実行環境**:

- **GitHub Actions**: `.github/workflows/wholesale.yml`（実装済）
  - 月次 cron: 毎月21日 16:30 JST 自動実行（前月分を取得）
  - 手動 dispatch: `mode=backfill` で過去全月一括 / `mode=incremental` で未取得月のみ / `month=YYYYMM` で特定月 / `force` で既存上書き
  - 変更があれば bot が自動 commit & push（main ブランチ）
- **ローカル**: Windows/Mac/Linux でも同スクリプトを実行可能
- Claude Code on the web 環境は東京都サイトへ到達不可（host_not_allowed）。`--file` モード以外は GitHub Actions かローカルで実行する

### 1.8 推測値運用魚種の再判定

`fish-species-map.json` で `geppo_item: null` の9種（アブラボウズ・アカムツ・アラ・カイワリ・カンコ・キントキ・モロコ・シロアマダイ・ハゼ）は **2026/04 単月での判定**。

旬で月報に現れる可能性があるため、**バックフィル完了後（202502〜直近 全15か月）に再判定** する:

1. `wholesale-prices.json` の `prices[].price_fish_id == null` 行を全月で集約
2. 該当魚種のエイリアス（aliases）と品目名で照合
3. 単独項目が見つかれば `fish-species-map.json` の `geppo_item` を埋め直す

旬で出る可能性が高い順:
- アカムツ（夏〜秋）・キントキ（夏）・カイワリ（夏〜秋）・カンコ（春〜夏）・アラ（冬〜春）

構造的に出ない可能性が高い順:
- シロアマダイ（希少すぎ）・モロコ・ハゼ・アブラボウズ

## 2. 魚種対比表（JSONに移管）

魚種ごとの完全な対比表は **`fish-species-map.json`** を SoT とする。本MDは構造と運用ルールのみ記述。

### 2.1 ファイル構成

```
docs/fish-value/fish-species-map.json
```

- 1ファイルで target（サイト掲載 64魚種）＋ bycatch（外道 8魚種）を統合管理
- 計 72エントリ

### 2.2 エントリのスキーマ

| フィールド | 型 | 説明 |
|---|---|---|
| `site_fish_id` | str / null | `docs/fish/{site_fish_id}.html` ファイル名と完全一致。外道は null |
| `site_display_name` | str | サイト表示名 |
| `price_fish_id` | str | 価格マスタ参照キー。同一魚種は共有（例: アジ・ビシアジ → maaji）|
| `geppo_item` | str / null | 月報での品目名。null = 月報単独項目なし（推測値運用）|
| `geppo_category` | str / null | 月報「大分類/中分類」 |
| `aliases` | str[] | 別名・地方名・サイズ違い名。価格計算では未使用・検索/UI用 |
| `category` | "target" / "bycatch" | サイト掲載／外道 |
| `input_modes` | str[] | `["cm"]`（小型・cm入力）or `["kg"]`（大型・kg入力）|
| `price_derived_from` | object / null | 他魚種から導出する場合の倍率定義 |
| `note` | str | 補足 |

### 2.3 統計（fish-species-map.json v1 確定）

| 区分 | 件数 | 内訳 |
|---|---|---|
| target | 64 | docs/fish/*.html 全件と一致 |
| bycatch | 8 | サバ・マイワシ・ウルメイワシ・カタクチイワシ・ソウダガツオ・アンコウ・ボラ・コノシロ |
| 月報直接マッチ | 55 | 月報品目名が `geppo_item` に確定 |
| 推測値運用 | 9 | アブラボウズ・アカムツ・アラ・カイワリ・カンコ・キントキ・モロコ・シロアマダイ・ハゼ |
| 価格導出（derived）| 1 | シロアマダイ = アマダイ × 2.0〜3.0倍 |

### 2.4 アカムツ＝ノドグロ

標準和名「アカムツ」、通称「ノドグロ」（喉が黒い）。北陸・山陰での呼称。`aliases` に両方記載。

**月報にはどちらも単独項目がなく「その他鮮魚」に集約** されているため、推測値運用とする。市場相場としては「むつ」（クロムツ）2,381円/kgより明確に高い（5,000〜10,000円/kg帯）。`fish-price-master.json` で個別定義する。

### 2.5 シロアマダイの価格導出

シロアマダイは超高級魚で、市場でアカアマダイの2〜3倍の単価で取引される。

```json
"price_derived_from": {
  "base_price_fish_id": "amadai",
  "ratio_low": 2.0,
  "ratio_high": 3.0
}
```

計算ロジック:

```
シロアマダイ卸値 = アマダイ卸値 × ratio_low 〜 ratio_high
                = 4,788 × 2.0 〜 4,788 × 3.0
                = 9,576 〜 14,364 円/kg
```

将来、月報に「しろあまだい」品目が独立して入った場合は `geppo_item` を埋めて `price_derived_from` を null にする。

### 2.6 外道（bycatch）の扱い

外道は サイト個別ページ無し・釣果報告で頻繁に出る副産物。価格計算は target と同じロジック。

| display_name | geppo_item | 平均円/kg（2026/04）|
|---|---|---|
| サバ | さば | 655 |
| マイワシ | まいわし | 452 |
| ウルメイワシ | うるめいわし | 288 |
| カタクチイワシ | かたくちいわし | 602 |
| ソウダガツオ | そうだかつお | 688 |
| アンコウ | あんこう | 929 |
| ボラ | ぼら | 319 |
| コノシロ | このしろ | 2,528 |

UIではターゲット魚種選択時に「外道も入力する」セクションを開閉できるようにする（デザイン.md 参照）。

### 2.7 魚種追加時の手順（唯一の接点）

新魚種が `docs/fish/` に追加されたとき:

1. `fish-species-map.json` に新エントリを追加
2. 月報品目を実データで照合し `geppo_item` を確定
3. なければ `note: "推測値運用"` を明記
4. 必要なら `fish-price-master.json` に推測単価を追加（PR2 以降）
5. アプリ側コード変更不要（JSON駆動）

これ以外の同期作業は発生しない。

## 3. 倍率テーブル（卸値 → 小売値）

### 3.1 設計方針

魚種カテゴリ別の **base 倍率** × サイズ帯別の **大物プレミアムブースト** の2段構造。

```
小売換算 = 卸売換算 × category_base × size_class_mult
```

`fish-price-master.json` の以下を参照:

- `category_definitions[カテゴリ].base_low / base_high`
- `size_class_multipliers[size_class]`
- `prices[pfid].size_bands[i].wholesale_low/high` `retail_low/high`（事前計算済み・直接使う）

### 3.2 倍率レンジ（業界一般値・甲案採用）

実証検証: `tmp_retail_verify.md`（小売物価統計 vs 月報最新月 実倍率 8魚種で 1.4〜3.4 に分布・業界一般値と整合）。

| カテゴリ | 例 | base_low | base_high |
|---|---|---|---|
| 大衆魚 | アジ・サバ・イワシ・キス | 2.0 | 3.0 |
| 中級魚 | マダイ・カワハギ・ヒラメ・イサキ・カレイ | 1.8 | 2.5 |
| 高級魚 | アカムツ・クロムツ・キンメ・アマダイ・トラフグ・シロアマダイ・アラ | 1.6 | 2.2 |
| 青物 | カツオ・ブリ系・キハダ・カンパチ・サワラ・タチウオ・シーバス | 1.7 | 2.4 |
| イカタコ | スルメイカ・ヤリイカ・アオリイカ・マダコ・コウイカ | 1.8 | 2.6 |

### 3.3 大物プレミアムブースト

サイズ帯ごとに size_class を持ち、卸→小売倍率をブースト:

| size_class | 倍率ブースト | 用途 |
|---|---|---|
| small | × 0.95 | 業界一般値より控えめ |
| standard | × 1.00 | 業界一般値そのまま（基準点） |
| large | × 1.20 | 大物プレミアム +20% |
| premium | × 1.45 | 特大プレミアム +45% |

例（マダイ）:
- 標準 (1.5kg以下): 卸 854 × 1.8〜2.5 × **1.00** = 小売 1,537〜2,135 円/kg
- 特大 (3.0kg超): 卸 1,800 × 1.8〜2.5 × **1.45** = 小売 4,698〜6,525 円/kg

### 3.4 加工度方針（Phase 1）

**「丸ごと小売換算」のみ表示する**（柵・切り身の加工済み価格は出さない）。

理由:
- 釣果は「丸ごと」で持ち帰るため、丸ごと小売価格との比較が直感的
- スーパーの柵・切り身は歩留まり 30-50% の加工が入っており、卸売丸ごとの 4-7倍に跳ね上がる（小売物価統計の刺身用さくマダイは卸の 7.7倍）
- 「自分でさばけば XXX 円相当」の追加表示は Phase 2 以降の発展論点

### 3.5 出典の明記

`fish-price-master.json` の各エントリに `wholesale_source` `data_basis` を必須にする。

## 4. サイズ・重量による単価変動

### 4.1 月報のサイズ別データ事情

**月報は基本的にサイズ別の単価データを持たない**（数量・金額のみ）。ただし以下の系統では **「サイズで品目が分かれている」** ので実質的にサイズ別データとして使える:

| 系統 | サイズ別品目 | 単価（2026/04・円/kg）|
|---|---|---|
| ブリ系 | わかし → いなだ → わらさ → ぶり → ひらまさ | 422 → 595 → 579 → 649 → 1,060 |
| タイ系 | かすご → ちたい → またい（天然）→ あまたい | 622 → 760 → 854 → 4,788 |
| マグロ系 | めじ → きめじ → きわだ（国内）→ まぐろ（国内）| 1,262 → 657 → 1,212 → 3,501 |
| カツオ系 | そうだかつお → かつお → すまかつお | 688 → 908 → 2,073 |

→ ブリ系・タイ系・マグロ系の釣果は **入力サイズに応じて `price_fish_id` を自動切替** する設計が可能。

### 4.2 一般魚種のサイズ単価補正

ブリ系等の系統に乗らない魚種（アジ・カワハギ・タチウオ等）は月報からはサイズ別単価が取れない。`fish-price-master.json` で `size_weight_curve` + `price_bands` を持つ:

- `size_weight_curve`: cm → kg の推定曲線（離散点の配列）
- `price_bands`: kg帯ごとの卸値レンジ（推測値）

### 4.3 cm入力時の3段フロー（一般魚種）

```
[入力] 平均サイズ 25cm × 35匹
   ↓
[1] size_weight_curve で 25cm → 0.16kg/尾
   ↓
[2] 月報単価 または price_bands で単価決定（円/kg）
   ↓
[3] 推定総重量 5.6kg × 単価 = 卸売換算
              × 倍率 retail_multiplier = 小売換算
```

### 4.4 ブリ系等のサイズ→品目切替

```
[入力] 平均重量 2.0kg × 1匹（ブリ系）
   ↓
[1] サイズ閾値判定: 2.0kg → わらさ帯（1.5〜3.0kg）
   ↓
[2] 月報「わらさ」単価 579円/kg
   ↓
[3] 卸 2.0kg × 579 = 1,158円 × 倍率1.7〜2.4 = 小売 1,969〜2,779円
```

サイズ閾値は `fish-price-master.json` で魚種系統ごとに定義。

### 4.5 kg入力時のフロー（大型魚）

cm→kg 変換ステップをスキップし、直接 price_bands or サイズ→品目切替に進む。

### 4.6 価格参照の順序（実装簡素化・2026/05/24 確定）

`fish-price-master.json` の `prices[pfid].size_bands[i]` を直接参照する。月報の参照は `generate_price_master.py` の段階で完了済み（事前計算）。

```
1. 入力（魚種・匹数・サイズcm/重量kg）
2. size_weight_curve で kg 換算（cm入力時のみ）
3. 入力 kg が該当する size_bands[i] を選択（kg_max 昇順）
4. その band の wholesale_low/high・retail_low/high を直接使う
5. 推定総重量 = 入力 kg × 匹数
6. 卸売換算 = 推定総重量 × wholesale_low/high
7. 小売換算 = 推定総重量 × retail_low/high
```

データ品質的フォールバック（roadmap）:
- 月報の最新月に該当 geppo_item が無くなった場合 → 前月版 fish-price-master.json を保持
- 推測値運用 9魚種（abura-bouzu / akamutsu / ara / kaiwari / kanko / kintoki / moroko / shiro-amadai / haze）は fallback_wholesale_* を経験値で持つ
- 月報に独立項目が将来追加されたら fish-species-map.json の `geppo_item` を埋め直して `generate_price_master.py` 再実行

## 6. データ品質ルール

| ルール | 内容 |
|---|---|
| 単一価格断定禁止 | 必ずレンジ（low〜high）で持つ |
| 出典明記 | 各エントリに `source_note` または `note` |
| 更新日明記 | 各JSONに `updated_at` |
| 推測値の明示 | 一次データが無い場合は `note: "推測値運用"` を必須 |
| 倍率の極端値禁止 | `retail_multiplier_high / low ≤ 2.0` を守る |
| 天然優先 | 月報複数候補は天然＞活魚＞養殖の順 |
| 時系列保持 | `wholesale-prices.json` は月次データを上書きせず追記 |
