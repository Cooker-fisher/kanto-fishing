## §2 ユーザー判断（確定値）

### 判断 1: 主指標の方向 → **promise_break 方向（低いほど良い）**

確定値: composite_promise_break として記録（既存 cnt/size/kg と方向統一）。

実装影響:
- 主指標カラム: `promise_break_rate` に composite 用の値を格納（既存スキーマ流用）
- 数式: `1 - 加重平均(is_*_in_range)` または直接 `加重平均(is_*_below_lo)`
- クエリ時に他 metric と単一クエリで比較可能

### 判断 2: cnt のみコンボの扱い → **出す + component_count 列追加**

確定値: cnt のみコンボ（size/kg ともに NULL）にも composite 値を記録。ただし「何個の指標で計算したか」を `component_count` 列で明示する。

実装影響:
- combo_range_backtest の CREATE TABLE に `component_count INTEGER` 列を追加（§3.2 詳細）
- composite 行のみ `component_count` を 1/2/3 で記録:
  - 1: cnt のみ
  - 2: cnt + size または cnt + kg
  - 3: 全揃い（cnt + size + kg）
- 既存 cnt/size/kg 行は `component_count = NULL`（8 要素タプルの末尾に None を追加 → §3.4 詳細）
- コンボ数: 最大 290（cnt 有効全コンボ）
- クエリ時に `WHERE component_count >= 2` で「複数指標統合」のみ抽出可能（domain 推奨）

### 判断 3: 副指標の保持範囲 → **promise_break + coverage + over_expect + winkler（4 軸全保存）**

確定値: cnt/size/kg と同じ 4 指標構造を保つ。クエリ用途を広く保ち、後の解析で柔軟性を確保。

実装影響:
- composite 行も既存スキーマの 4 指標列（promise_break_rate / coverage / over_expect_rate / winkler）を全て保存
- bowzu_rate は composite では NULL（cnt のみ計算可能・統合する意味薄い）
- 計算方法:
  - composite_promise_break = 行レベル加重平均（is_below_lo の重み付き平均を全行で平均）
  - composite_coverage = 行レベル加重平均（is_in_range の重み付き平均を全行で平均）
  - composite_over_expect = 行レベル加重平均（is_above_hi の重み付き平均を全行で平均）
  - composite_winkler = NULL（v1.2 では未実装・§8 リスク 7 参照）

### 判断 4: 採用/撤回判定方針 → **エラーさえ出なければ採用**

確定値: Phase C は新規行追加のみで既存 cnt/size/kg を壊さない。実装エラーなしで成功基準（§4）クリアなら採用確定。

実装影響:
- §4 の成功基準で十分（撤回基準 X pt は不要・新規メトリック追加のため）
- ロールバック計画は不要（既存値非破壊なので戻す必要がない）
- 全コンボ再実行後の判定は「成功基準全クリア → engineer コミット」の単純フロー

### 技術確定事項（ユーザー判断不要・既存実装との整合性ルール）

**cnt 評価軸: actual_avg ベース（既存単体 cnt 評価と統一）**

実コード L3636-3658 の cnt ループは `act_avg = d["cnt_avg"][1]`（actual_avg）で評価している。composite の cnt 成分も同じ軸を使うことで、cnt 単体評価と整合する。

- cnt 成分の `is_cnt_below = 1.0 if act_avg < pred_lo else 0.0`（actual_avg ベース）
- cnt 成分の `is_cnt_above = 1.0 if act_avg > pred_hi else 0.0`（actual_avg ベース）

size（L3704-3713）/ kg（L3762-3767）は actual_min/max ベースで評価している。composite の size/kg 成分も同じ軸を維持する。

### 3.2 CREATE TABLE 変更と needs_recreate 拡張

**ALTER TABLE は使わない。** CREATE TABLE への列追加 + `needs_recreate` 判定拡張の方式を採用する。

理由: ALTER TABLE は実行タイミング（どの関数内で実行するか）が曖昧になりやすく、既存の `needs_recreate` ロジック（DROP→CREATE）との併存が複雑になる。CREATE TABLE 列追加 + `needs_recreate` 拡張で一度の DROP→CREATE により全行が新スキーマになる。

**変更箇所 1: `save_range_backtest()` 内の CREATE TABLE 文（L4565-4580 相当）**

```python
# 変更前 (PRIMARY KEY 行の後・winkler と n の間に component_count がない)
conn.execute("""
    CREATE TABLE IF NOT EXISTS combo_range_backtest (
        fish               TEXT,
        ship               TEXT,
        metric             TEXT,
        horizon            INTEGER,
        promise_break_rate REAL,
        over_expect_rate   REAL,
        coverage           REAL,
        bowzu_rate         REAL,
        winkler            REAL,
        n                  INTEGER,
        updated_at         TEXT,
        PRIMARY KEY (fish, ship, metric, horizon)
    )
""")

# 変更後（component_count INTEGER 列を updated_at の直前に追加）
conn.execute("""
    CREATE TABLE IF NOT EXISTS combo_range_backtest (
        fish               TEXT,
        ship               TEXT,
        metric             TEXT,     -- "cnt" / "size" / "kg" / "composite"
        horizon            INTEGER,
        promise_break_rate REAL,
        over_expect_rate   REAL,
        coverage           REAL,
        bowzu_rate         REAL,
        winkler            REAL,
        n                  INTEGER,
        component_count    INTEGER,  -- composite のみ 1/2/3。cnt/size/kg は NULL
        updated_at         TEXT,
        PRIMARY KEY (fish, ship, metric, horizon)
    )
""")
```

**変更箇所 2: `save_range_backtest()` 内の `needs_recreate` 判定（L4555-4560 相当）**

```python
# 変更前
needs_recreate = (
    existing_cols and (
        "metric" not in existing_cols
        or "max_over_rate" in existing_cols

)

# 変更後（component_count がなければ再作成）
```

この変更により、既存の analysis.sqlite には `component_count` がないため DROP→CREATE が自動実行される。既存 cnt/size/kg 行は再実行で復元される（run_full_deepdive.py 実行時に全コンボが再計算される前提）。

### 3.3 composite 計算ブロック（実コードへの挿入位置と実装）

**挿入位置**: L3779 の `range_bt_data.append(("kg", ...))` の直後・L3781（star_backtest セクション）の前。

実コードの確認済みキー構造（`_range_by_key[H][_rk]`）:
- `d["cnt_min"]` → `(pred_lo, act_min)` タプル（cnt_min モデルの pred と actual_cnt_min）
- `d["cnt_max"]` → `(pred_hi, act_max)` タプル（cnt_max モデルの pred と actual_cnt_max は未使用）
- `d["cnt_avg"]` → `(pred_avg, act_avg)` タプル（cnt_avg モデルの pred と actual_avg）
- `d["size_avg"]` → `(pred_c, act_avg_size)` タプル（size_avg モデルの pred と actual_size_avg）
- `d["_size_min_act"]` → スカラー（actual_size_min）
- `d["_size_max_act"]` → スカラー（actual_size_max）
- `d["_decade_no"]` → スカラー（旬番号）
- `d["kg_avg"]` → `(pred_c, act_avg_kg)` タプル
- `d["_kg_min_act"]` → スカラー（actual_kg_min）
- `d["_kg_max_act"]` → スカラー（actual_kg_max）

size/kg の pred_lo/pred_hi は `_range_by_key` に格納されていない（各メトリックループ内でローカル計算）。composite ブロック内で再計算する。`_bt_decadal_size_ratio` / `_bt_decadal_kg_ratio` は `deep_dive()` 関数スコープ内の変数として利用可能。

```python
    # ── composite メトリック（Phase C: 加重平均 hit_rate）─────────────────────────
    # 設計根拠: plan_C_2026-05-08.md §3 / plan_hit_rate_2026-05-08.md §C-1〜C-4
    # 補遺3 整合: composite は内部評価のみ。HTML 表示禁止（§8 リスク 3 参照）。
    #
    # 重み（plan_C §1.4 確定値）:
    #   W_CNT = 0.6 / W_SIZE = 0.3 / W_KG = 0.1
    #
    # NULL 時の重み再正規化:
    #   有効メトリックの重みのみで正規化（0 除算ガード必須）
    #   cnt は必須（cnt なし → composite 計算不可・行スキップ）
    #
    # 評価軸の統一（§2 技術確定事項）:
    #   cnt  → actual_avg ベース（実コード L3636-3658 の cnt ループと統一）
    #   size → actual_min/max ベース（Phase B-α' L3704-3713 と統一）
    #   kg   → actual_min/max ベース（Phase B-β-4 L3762-3767 と統一）
    #
    # フラグの排他性（数学的保証）:
    #   cnt:  is_cnt_below + is_cnt_above + is_cnt_in = 1
    #         （actual_avg は 1 点なので below / above / in は完全排他的）
    #   size: is_size_below + is_size_above + is_size_in <= 2
    #         （actual_min < pred_lo かつ actual_max > pred_hi の場合 below=1, above=1 になり得る）
    #         （区間が pred 区間を完全に外から囲むケース）
    #   kg:   size と同様 <= 2
    #   ※ size/kg は「一方の端だけ外れる」場合 below=0, above=0, in=0 も有り得る
    #      （actual_min >= pred_lo かつ actual_max > pred_hi = above=1, in=0）
    #
    # 行レベル集計採用の根拠:
    #   各釣行（行）単位で cnt/size/kg の同時達成度を加重平均し、全行で再平均する方式。
    #   テーブルレベル集計（metric ごとの promise_break_rate を加重平均）では
    #   「cnt が良くて size が悪いコンボ」の悪化を過小評価するリスクがある。
    #   行レベルは「全指標が同時に範囲内に入った率」という統合評価の意味を保つ。
    #   具体例: cnt が良好(0%)・size が悪い(100%) の 1 行コンボの場合、
    #     テーブルレベル(0.667×0 + 0.333×1 = 0.333)よりも
    #     行レベル(W_CNT×0 + W_SIZE×1)/w_sum = 0.333 で一致するが、
    #     複数行が混在するコンボでは、cnt が良く size が悪い日と cnt が悪く size が良い日が
    #     混在する場合にテーブルレベルではこの日間相関構造が失われる。
    #     行レベルでは各日の同時達成を評価できるため、統合評価として適切。
    #
    # 行レベルとテーブルレベル（v1.5 修正・stat-reviewer 指摘反映）:
    #   現実装は線形加重和のため、行レベル集計とテーブルレベル集計は数学的に等価。
    #   linearity of expectation により以下が成立:
    #     sum_i(W_CNT×f_cnt_i + W_SIZE×f_size_i) / (n×w_sum)
    #     ≡ W_CNT/w_sum × mean(f_cnt_i) + W_SIZE/w_sum × mean(f_size_i)
    #   §5.2 ステップ 3 の比較 SQL で「両者が一致する」のが期待値。差があれば実装ミス。
    #
    #   行レベル構造を採用する根拠: 将来の非線形統合指標
    #     （例: cnt AND size の同時達成率 = is_cnt_in × is_size_in の積）
    #     への拡張可能性を確保するため。現時点では数学的に同値だが、
    #     非線形化する将来拡張に備えて行レベルアーキテクチャを維持する。
    #
    # composite の is_cnt_above 定義について:
    #   composite の is_cnt_above は「act_avg > pred_hi」（単純超過ベース）を使用。
    #   cnt 単体の over_expect_rate（実コード L3645-3646: pred_hi/act_avg > 2.5 比率ベース）とは
    #   定義が異なる。composite は内部評価のみのため、単純超過ベースで許容。
    W_CNT, W_SIZE, W_KG = 0.6, 0.3, 0.1

    for H in HORIZONS:
        composite_rows = []
        for _rk, d in _range_by_key[H].items():

            # ── cnt（必須）: actual_avg ベース ──────────────────────────────────
            # L3636-3658 の cnt ループと評価軸を統一
            if "cnt_min" not in d or "cnt_max" not in d or "cnt_avg" not in d:
                continue
            pred_lo_cnt, _ = d["cnt_min"]   # cnt_min モデルの pred 値
            pred_hi_cnt, _ = d["cnt_max"]   # cnt_max モデルの pred 値
            _,    act_avg  = d["cnt_avg"]   # actual_avg（cnt 評価軸）
            if pred_lo_cnt is None or pred_hi_cnt is None or act_avg is None:
                continue
            # cnt フラグは完全排他的（actual_avg は 1 点なので is_below + is_above + is_cnt_in = 1）
            # ※ is_cnt_above は act_avg > pred_hi の単純超過ベース。
            #    cnt 単体 over_expect（pred_hi/act_avg > 2.5 比率ベース L3645-3646）とは定義が異なる。
            #    composite は内部評価のみのため、単純超過ベースで許容（補遺3 整合）。
            is_cnt_below = 1.0 if act_avg < pred_lo_cnt else 0.0
            is_cnt_above = 1.0 if act_avg > pred_hi_cnt else 0.0
            is_cnt_in    = 1.0 if pred_lo_cnt <= act_avg <= pred_hi_cnt else 0.0

            # ── size（任意）: actual_min/max ベース、pred は再計算 ───────────────
            # Phase B-α' L3677-3726 の size ループと評価軸を統一
            # pred_lo/pred_hi は _range_by_key に未格納のため composite ブロックで再計算
            is_size_below = is_size_above = is_size_in = None
            if "size_avg" in d:
                pred_c_sz, _ = d["size_avg"]   # size_avg モデルの pred 値（点予測）
                act_sz_min   = d.get("_size_min_act")
                act_sz_max   = d.get("_size_max_act")
                dec_no       = d.get("_decade_no")
                if (pred_c_sz is not None
                        and act_sz_min is not None and act_sz_max is not None):
                    _rmin_sz, _rmax_sz = _bt_decadal_size_ratio.get(
                        dec_no, (_global_ratio_min_p20, _global_ratio_max_p80)
                    )
                    pred_lo_sz = pred_c_sz * _rmin_sz
                    pred_hi_sz = pred_c_sz * _rmax_sz
                    # size フラグは <= 2（act_min < pred_lo かつ act_max > pred_hi で below=above=1 になり得る）
                    is_size_below = 1.0 if act_sz_min < pred_lo_sz else 0.0
                    is_size_above = 1.0 if act_sz_max > pred_hi_sz else 0.0
                    is_size_in    = (1.0 if (act_sz_min >= pred_lo_sz
                                             and act_sz_max <= pred_hi_sz) else 0.0)

            # ── kg（任意）: actual_min/max ベース、pred は再計算 ─────────────────
            # Phase B-β-4 L3737-3779 の kg ループと評価軸を統一
            is_kg_below = is_kg_above = is_kg_in = None
            if "kg_avg" in d:
                pred_c_kg, _ = d["kg_avg"]   # kg_avg モデルの pred 値（点予測）
                act_kg_min   = d.get("_kg_min_act")
                act_kg_max   = d.get("_kg_max_act")
                dec_no_kg    = d.get("_decade_no")  # _decade_no は size と共通キー
                if (pred_c_kg is not None
                        and act_kg_min is not None and act_kg_max is not None):
                    _rmin_kg, _rmax_kg = _bt_decadal_kg_ratio.get(
                        dec_no_kg, (_global_kg_ratio_min_p20, _global_kg_ratio_max_p80)
                    )
                    pred_lo_kg = pred_c_kg * _rmin_kg
                    pred_hi_kg = pred_c_kg * _rmax_kg
                    # kg フラグは <= 2（size と同様）
                    is_kg_below = 1.0 if act_kg_min < pred_lo_kg else 0.0
                    is_kg_above = 1.0 if act_kg_max > pred_hi_kg else 0.0
                    is_kg_in    = (1.0 if (act_kg_min >= pred_lo_kg
                                           and act_kg_max <= pred_hi_kg) else 0.0)

            # ── 重み再正規化 ─────────────────────────────────────────────────────
            w_sum = W_CNT
            component_count = 1   # cnt は必須なので 1 から始まる
            if is_size_in is not None:
                w_sum += W_SIZE
                component_count += 1
            if is_kg_in is not None:
                w_sum += W_KG
                component_count += 1

            if w_sum <= 0:
                continue   # 防御的ガード（理論上 w_sum >= 0.6 のはず）

            # ── 4 指標を行レベル加重平均（正規化済み）───────────────────────────
            below_w = W_CNT * is_cnt_below
            above_w = W_CNT * is_cnt_above
            in_w    = W_CNT * is_cnt_in
            if is_size_in is not None:
                below_w += W_SIZE * is_size_below
                above_w += W_SIZE * is_size_above
                in_w    += W_SIZE * is_size_in
            if is_kg_in is not None:
                below_w += W_KG * is_kg_below
                above_w += W_KG * is_kg_above
                in_w    += W_KG * is_kg_in

            composite_rows.append({
                "below": below_w / w_sum,
                "above": above_w / w_sum,
                "in":    in_w    / w_sum,
                "comp":  component_count,
            })

        n_comp = len(composite_rows)
        if n_comp < MIN_N_RANGE:
            continue

        # ── 4 指標の集計（行レベル平均）─────────────────────────────────────────
        comp_pb  = sum(r["below"] for r in composite_rows) / n_comp
        comp_oe  = sum(r["above"] for r in composite_rows) / n_comp
        comp_cov = sum(r["in"]    for r in composite_rows) / n_comp
        # winkler は cnt/size/kg で単位が異なるため加重平均は意味をなさない → NULL
        comp_winkler = None

        # component_count 代表値: 平均値の四捨五入（期待値の整数化・実態の近似値）
        # 注意: 1 と 3 が均等混在するコンボでは四捨五入結果は 2 になるが最頻値は不定。
        #       同一コンボ内で行ごとに component_count が異なる場合（kg が一部行のみ存在等）、
        #       代表値は行間の分布を潰す。層別解析が必要な場合は §8 リスク 8 参照。
        avg_comp = sum(r["comp"] for r in composite_rows) / n_comp
        comp_count_repr = round(avg_comp)

        range_bt_data.append((
            "composite", H, comp_pb, comp_oe, comp_cov, None, comp_winkler, n_comp,
            comp_count_repr   # 9 要素目: component_count（save_range_backtest で受け取る）
        ))
```

**winkler の取り扱い**: cnt/size/kg の winkler は単位が異なる（cnt は匹ベース区間幅、size は cm ベース、kg は kg ベース）。単純加重平均は単位混在で意味をなさないため NULL とする。後の Phase で行レベルスコアを無次元化してから加重平均する実装は可能（リスク 7 参照）。

## §8 リスクと対策

### リスク 1: 既存 cnt/size/kg 行の値が壊れる

**症状**: composite 計算ブロックが `_range_by_key` を変更し、後続処理で欠損

**対策**: composite ブロックは `_range_by_key[H]` を読み取りのみで参照。reviewer が write 操作の不在を確認

**検証**: 成功基準 4・8（既存行数・数値・component_count NULL の確認）

### リスク 2: NULL 重み再正規化のバグ（除算 0・指標欠損）

**症状**: `w_sum = 0.0` で除算実行・ZeroDivisionError

**対策**: `if w_sum <= 0: continue` ガード実装。cnt が None の行は composite 対象外（理論上 w_sum >= 0.6）

**検証**: 1 コンボ動作確認で composite_val が 0〜1 に収まることを確認

### リスク 3: HTML 表示で composite を誤って表示してしまう（補遺3 違反）

**症状**: 実装者が composite を crawler.py または forecast HTML に追加してしまう

**対策**:
- 本 Plan §1.3 に「HTML 表示禁止」を明記
- engineer コミット前に `grep -n "composite" crawler.py` で参照不在確認

**補遺3 再確認**: composite はユーザー表示対象ではない。analysis.sqlite 内部評価のみ

### リスク 4: 重み固定の妥当性（kg NULL 率 71% での実質値）

**症状**: kg 有効コンボ少なく（103 コンボ）、composite 計算に kg 寄与少ない

**対策**: 設計上の意図は「kg 記録がある場合のみ加点」で正しい動作。実質 cnt+size 構成（0.667/0.333）になることは許容（plan_hit_rate §C-4 既知）

### リスク 5: `_range_by_key` キー名不一致（v1.2 で解消）

**v1.1 指摘（analyst CRITICAL 1）**: 存在しないキー（`_cnt_pred_lo` 等）を参照していた。

**v1.2 修正**: 実コードの実際のキー構造（§3.3 の確認済みキー一覧）を反映。タプルアンパックで正しく取得。

**検証**: 実装前に 1 コンボで `print(f'keys: {list(d.keys())}')` を実行してキー確認（CLAUDE.md ルール）

### リスク 6: component_count 列追加で既存クエリが影響を受ける

**症状**: 既存の `SELECT * FROM combo_range_backtest` クエリが新列を読み込み、後続処理で予期しない動作

**対策**: DROP→CREATE による再作成後、全コンボ再実行で全行が新スキーマで統一される。既存クエリは `SELECT *` でも新列を無視できる

**検証**: 全コンボ実行後に predict_count.py 等の caller が変わらず動作することを確認

### リスク 7: winkler を NULL にする判断の妥当性

**症状**: composite_winkler が常に NULL のため、後の解析で活用できない

**対策**: v1.2 では NULL 採用。理由: cnt の winkler は匹ベース区間幅、size は cm、kg は kg と単位が異なり加重平均は意味をなさない。無次元化（各メトリックの winkler を自身の scale で正規化）が必要。必要と判断した場合は Phase D で別実装。

### リスク 8: component_count 代表値の統計的限界（stat-reviewer MAJOR-2 指摘）

**症状**: 同一コンボ内で行ごとに component_count が異なる場合（kg が一部行のみ存在等）、代表値（平均の四捨五入）が行間変動を潰す。

**注意点**:
- component_count が行ごとに 1 と 3 が混在するコンボでは、代表値 2 が実態を反映しない（1 と 3 が均等混在時は四捨五入結果 2 になるが最頻値は不定）
- 層別解析（component_count=1/2/3 で promise_break が系統的に異なるか）は別途確認が必要
- ランキング集計時は `WHERE component_count >= 2` を推奨（domain 推奨・§2 判断 2 に記載）

**検証**: §9 Phase 6 の確認 SQL 4 で `component_count` の分布ヒストグラム（DISTINCT + COUNT）を確認する

### リスク 9: composite_coverage 著しく低い（domain MINOR-1 指摘・v1.3 閾値更新）

**症状**: composite_coverage が著しく低い場合、「全指標が同時に範囲内」の基準が厳しすぎる可能性がある

**対策**（2 段階閾値・v1.3 更新）:
- 0.10 未満: 異常値・バグ可能性として原因調査が必要（成功基準 9 の異常判定）
- 0.30 未満: 品質低下として cnt/size 単体評価を優先・Phase D で閾値変更を別途検討
- 既存 cnt(79.4%)/size(40.3%)/kg(42.2%) coverage 実績から 0.05 はほぼ壊れているレベル（v1.2 の閾値を更新）

### リスク 10: 全コンボ再実行が中断した場合の状態（v1.3 追加・v1.4 で `--reset-best` 明示）

**症状**: テーブルが空または不完全な状態になる可能性（ネットワーク断・プロセス強制終了等）

**対策**:
- needs_recreate は既存列を検査するため、2 回目以降の実行で DROP しない（component_count が存在すれば再作成しない）
- 再実行コマンド例: `python analysis/V2/methods/run_full_deepdive.py --workers 4 --reset-best`

- cnt/size/kg/composite 行は実行済みコンボ分から追記されるため、再実行で未実行コンボ分が補完される

```sql
-- 確認 SQL 1: metric='composite' 行の追加確認(coverage 著しく低い場合は要確認)
SELECT metric, COUNT(*) as n, AVG(promise_break_rate) as avg_pb,
       AVG(coverage) as avg_cov, AVG(over_expect_rate) as avg_oe
FROM combo_range_backtest
WHERE horizon = 0
GROUP BY metric ORDER BY metric;

-- 確認 SQL 2: cnt/size/kg 行数が変わっていない(前後比較)
SELECT metric, COUNT(*) as n
FROM combo_range_backtest
WHERE horizon = 0 AND metric IN ('cnt', 'size', 'kg')
GROUP BY metric;

-- 確認 SQL 3: composite 値の範囲確認
SELECT MIN(promise_break_rate), MAX(promise_break_rate),
       MIN(coverage), MAX(coverage)
FROM combo_range_backtest
WHERE metric = 'composite';
-- 期待: すべて 0.0〜1.0 の範囲

-- 確認 SQL 4: component_count 分布ヒストグラム(stat-reviewer 指摘・代表値の妥当性確認)
SELECT component_count, COUNT(*) as n
FROM combo_range_backtest
WHERE metric = 'composite' AND horizon = 0
GROUP BY component_count
ORDER BY component_count;
-- 期待: 1/2/3 のみ・他の値なし・分布の偏りを目視確認

-- 確認 SQL 5: 既存 cnt/size/kg 行の component_count NULL 確認
SELECT COUNT(*) FROM combo_range_backtest
WHERE metric IN ('cnt', 'size', 'kg') AND component_count IS NOT NULL;
-- 期待: 0

-- 確認 SQL 6（domain 推奨）: ランキング用・component_count >= 2 絞り込み確認
SELECT fish, ship, promise_break_rate, coverage, component_count
FROM combo_range_backtest
WHERE metric = 'composite' AND horizon = 0 AND component_count >= 2
ORDER BY promise_break_rate
LIMIT 10;

-- 確認 SQL 7（成功基準 6 対応）: PRIMARY KEY 重複確認
-- v1.4: DISTINCT キーに区切り文字 '|' を追加し境界衝突を回避
SELECT COUNT(*) as total,
       COUNT(DISTINCT fish || '|' || ship || '|' || CAST(horizon AS TEXT)) as distinct_n
FROM combo_range_backtest
WHERE metric = 'composite';
-- 期待: total = distinct_n（完全一致）
```
