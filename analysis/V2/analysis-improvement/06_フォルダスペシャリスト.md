# 06_フォルダスペシャリスト（構成整理・運用設計）

## 役割

analysis/配下の**フォルダ構成を運用しやすく整理**する。  
特に「研究・改善スクリプト」と「毎日の実稼働スクリプト」の分離を担当。

---

## 確定分離案（2026/04/12）

### 問題

`analysis/V2/methods/` に研究用と実稼働用スクリプトが混在しており、  
crawl.yml が呼ぶスクリプトとそうでないものが一目でわからない。

### 推奨フォルダ構成

```
analysis/
├── README.md
├── analysis_config.py
├── run.py
├── V1/               ← アーカイブ（参照専用）
└── V2/               ← 現行
    ├── README.md
    ├── analysis-improvement/   ← このプロジェクト
    │   ├── 00〜07_*.md
    │   ├── 90_決定ログ.md
    │   └── README.md
    ├── research/     ← ★ 純粋分析（手動実行・研究・改善）
    │   ├── combo_deep_dive.py      # 釣果×気象相関分析（主力）
    │   ├── backtest.py             # インサンプル精度テスト
    │   ├── backtest_oos.py         # アウトオブサンプルテスト
    │   ├── area_analysis.py        # エリア×魚種季節分析
    │   ├── compare_granularity.py  # 日次vs週次比較
    │   ├── season_analysis.py      # シーズン分析
    │   ├── season_detail.py        # シーズン詳細
    │   ├── ship_peaks.py           # 船宿別ピーク分析
    │   ├── analysis_combos.py      # 多次元原因分析
    │   ├── run_full_deepdive.py    # 全魚種並列実行ランチャー
    │   ├── parse_deepdive.py       # deep_dive → deepdive_params.json
    │   └── _paths.py               # パス自動解決ユーティリティ
    ├── production/   ← ★ 実稼働計算（crawl.yml が毎日自動実行）
    │   ├── predict_count.py        # 釣果数予測（D層）
    │   ├── risk_predict.py         # 出船リスク判定
    │   ├── weekly_analysis.py      # 週次集計
    │   ├── prediction_log.py       # 予測vs実績記録
    │   ├── enrich_catches.py       # 釣果CSV×気象結合
    │   ├── save_insights.py        # 分析結果DB保存
    │   └── cancel_threshold.py     # 欠航閾値計算
    ├── predict/      ← D層（prediction_log.py が使う予測関連）
    └── results/      ← 分析結果（移動しない）
        ├── analysis.sqlite         # メインDB（crawler.py が直接参照）
        ├── deepdive_params.json
        ├── deep_dive/  (380件)
        └── weekly/     (114件)
```

### 分類基準

| 分類 | フォルダ | 実行タイミング | 実行者 |
|------|---------|-------------|--------|
| 純粋分析・精度改善 | `research/` | 手動（改善時のみ） | 分析班 |
| 毎日の実稼働計算 | `production/` | 毎日自動（crawl.yml） | GitHub Actions |
| 分析結果DB | `results/analysis.sqlite` | 両方から書き込み | — |

### 移行手順

1. `methods/` → 各スクリプトを research/ か production/ に仕分け
2. `_paths.py` の `ROOT_DIR` を移行後パスに対応させる
3. `crawl.yml` の呼び出しパスを `production/` に更新
4. `methods/` フォルダは削除（空になったら）

### データインターフェース（不変）

- `analysis.sqlite` は `results/` に留まる。移動しない
- `crawler.py`（E層）は `analysis/V2/results/analysis.sqlite` を直接 sqlite3 で参照
- 中間 JSON は原則不要（04/09 確定）

---

## 運用ルール

| ルール | 内容 |
|--------|------|
| 1フォルダ=1責任 | research/ はスクリプト、production/ は実稼働 |
| crawl.yml に追加するスクリプトは production/ のみ | research/ のスクリプトは手動実行専用 |
| _paths.py インポート必須 | 移行後も `ROOT_DIR`, `DATA_DIR`, `RESULTS_DIR` を継続使用 |
| results/ 肥大化監視 | 現在499件。1000件超えたらサブフォルダ分割検討 |
