## 背景・段階分割の根拠

### 直前の診断結果（diag_size_2026-05-08.md）

数式で特定された真因:
```
現状:
  pred_lo = size_avg - MAE         (MAE 中央値 1.75cm)
  actual_min = actual_avg - range/2  (range/2 中央値 7.0cm)

→ pred_lo - actual_min = +5.25cm の系統バイアス（必然的・90.3% で正）
```

actual の min/max 幅が MAE の **4 倍** あるため、wMAPE 7.3%（中央誤差は良好）でも promise_break 53% は数式上避けられない。

### 段階分割の理由

Phase B 全体（cnt + size 拡張）を一括実装するリスク:
- 04/13「cnt_min 独立モデル BL-2 勝率 51.9%」で ratio 法に逃げた撤回経緯
- size でも同様のリスクがあり、cnt と size を一括変更すると失敗時のリバート範囲が大きい

→ size を先行実装し、成功確認後に cnt 拡張を行う段階分割を採用。

### 段階分割マップ

| Phase | 内容 | スコープ |
|---|---|---|
| **Phase B-α**（本 Plan） | size 単独独立予測 + 防御策 | 本セッション設計 / 次セッション実装 |

## スコープ

### 含むもの

1. **combo_deep_dive.py に size_min / size_max メトリック追加**
   - METRICS_LIST に "size_min", "size_max" を追加
   - section_backtest_rolling() 内で size_min / size_max の予測ループを cnt_min / cnt_max と同型で実装
   - rolling CV はリークなし（既存 leave-one-month-out CV を踏襲）

2. **combo_wx_params に独立パラメータ保存**
   - metric='size_min' / 'size_max' で因子・係数を保存
   - 過学習対策（適応的相関閾値・MAX_FACTORS=10・α_scale clip [0,1.2]）は既存と同じ

3. **predict_count.py の size_lo / size_hi を独立予測に切替**
   - 現状: `avg_size ± size_mae`
   - 変更後: `_apply_wx_correction(metric='size_min')` / `_apply_wx_correction(metric='size_max')` の独立予測値
   - フォールバック: 学習なしコンボは現状の `avg_size ± size_mae` 維持

4. **size 用防御策（floor=0.55 / clamp=1.0）追加**
   - floor: `pred_lo >= avg_size × 0.55`
   - clamp: `pred_hi <= pred_lo + avg_size × 1.0`
   - cnt 用（floor=0.3 / clamp=2.0）と倍率が大きく異なる点に注意

5. **combo_range_backtest 評価**
   - 現状: metric='size'（size_avg ベース）が記録されている
   - 拡張: metric='size'（既存維持）に加え、独立モデル後の判定ロジック（actual_size_min vs pred_lo_size 等）で再計算
   - 既存スキーマ（fish, ship, metric, horizon）はそのまま使用可

6. **combo_tuning/*.json に by_metric.size セクション追加**
   - 既存トップレベル（cnt 用）は維持（後方互換）
   - by_metric.size に size 用の adopted_factors / insights / wmape_h0 / promise_break_rate を保存

7. **全コンボ再実行（4時間）→ size promise_break_rate 改善確認**

## 数値根拠（diag_size_2026-05-08.md より）

### 防御策値の決定

| 防御策 | size 用推奨 | 根拠 |
|---|---|---|
| **floor: pred_lo >= avg × X** | **0.55** | size_min/avg の P5=0.581 に余裕（cnt 用 0.3 とは異なる） |
| **clamp: range <= avg × X** | **1.0** | actual range/avg の P90=0.732 をカバー（cnt 用 2.0 とは異なる） |

cnt 用倍率の流用は厳禁（過大クリップで予測が不自然になる）。

### 期待効果

- 構造バイアス +4.9cm が解消 → size promise_break 53% → **30% 以下**を見込む
- BL-2 勝率: 60-70% を見込む（cnt_min の 51.9% より構造的に改善・size_min は actual_min を直接学習するため）

### NULL 率対応

- size_min / size_max NULL 率 38.8%
- MIN_N_COMBO=30 のままだと size 学習サンプルが薄いコンボ多数
- → NULL 行除外後の n_valid を MIN_N_COMBO 判定に使う設計（Phase A の coverage 分母統一と同じパターン）

## 実装上のリスクと対策

### リスク 1: 多重共線性（size_min と size_max の相関 r=0.94）

- 同じ因子（spawn_season_n 等）を双方が採用 → 係数不安定
- 対策: 既存の MAX_FACTORS=10 制約 + 適応的相関閾値 + α_scale clip [0,1.2] を踏襲
- 検証: size_min と size_max の係数符号が逆転しないことを stat-reviewer に確認させる

### リスク 2: ratio 法の名残（学習側）

- combo_deep_dive.py L3285-3290 に cnt 用 ratio override がある
- size 系には ratio override が **無い**（既に独立学習されている前提）
- 念のため grep で size 系 ratio override の有無確認 → 実装前に必須

### リスク 3: 学習サンプル薄

- size NULL 率 38.8% で fold あたり n=15-30 程度になるコンボ多数
- 対策: フォールバック条件（学習無し/サンプル不足は旧 `avg_size ± size_mae`）を維持
- combo_wx_params に保存されないコンボは自動的にフォールバック動作

### リスク 4: combo_range_backtest の比較軸

- 現状の判定: actual_avg vs pred_lo（size_avg ベース）
- Phase B-α 後: actual_size_min vs pred_lo_size_min が自然な判定軸
- → metric='size_min' / 'size_max' の行を新設するか、既存 metric='size' を再定義するか要設計
- 推奨: 既存 metric='size' の判定式を「actual_min vs pred_lo」に変更（avg_size ベースを廃止）

### リスク 5: combo_tuning JSON の自動更新

- 342 ファイルへの size セクション追加コードは現状無い
- 既存の combo_tuning 更新ロジックを Read で確認してから設計
- 自動更新スクリプトは Phase B-α 実装と一緒に作成
