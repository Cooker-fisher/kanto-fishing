## A. 新規船宿コンボの精度実態

### A-1. コンボ分類

T32 で全コンボが `--reset-best` 付きで再実行されたため `updated_at` では T31 以前との区別ができない。
代替として ships.json の `chowari_id / sid` フィールドで分類した。

| カテゴリ | 定義 | combo_meta 内のコンボ数 |
|---|---|---|
| chowari 専用 | chowari_id あり・sid なし | 23件 |
| 両方あり | chowari_id + sid 両方あり | 37件 |
| fishing_v 専用 | chowari_id なし・sid あり | 53件 |
| ships.json 未登録 | どちらにも一致しない | 8件 |
| 合計 | | 384件・112船宿 |

chowari 専用 23件の内訳: 野毛屋釣船店 (5コンボ・旧来船宿) と、T32 で新規追加されたと思われる 18件 (増福丸・成銀丸・大松丸・村正丸・早川丸・正海丸・義心丸・鴨下丸kawana・魚磯丸・伝五郎丸・金沢八景黒川丸・黒川本家・深川吉野屋)。

### A-2. 精度比較（H=0, cnt_avg）

| 指標 | chowari 専用 (23件) | 既存コンボ (361件) | 差分 |
|---|---|---|---|
| wMAPE median | 37.69% (n=9) | 37.12% (n=352) | +0.57pt |
| wMAPE mean | 36.56% | 39.01% | -2.45pt |
| promise_break_rate composite median | 6.1% (n=7) | 14.0% (n=324) | **-7.9pt** (良好) |

**ボトルネック特定:**
- wMAPE 自体は既存コンボと同水準（精度悪化なし）
- BL-2 勝率が 6.8pt 低い = 旬別ベースライン (combo_decadal) に負けるケースが多い
- promise_break はむしろ低い（n が少ないため CI 幅が広い → 区間が外れにくい）
- n_records が少ない（median=37）ため、wMAPE の評価自体が不安定

### A-3. n_records 分布（MIN_N_COMBO=30 ギリギリ）

```
n<=40 (30~40):  15件 / 23件  (65.2%)
n<=50:          16件 / 23件  (69.6%)
n<=100:         19件 / 23件  (82.6%)
n>100:           4件 / 23件  (17.4%  ← 野毛屋釣船店のみ)
```

n=30~40 が 65% を占め、バックテスト月数が 2~4 か月程度しかないケースが多い。
適応的相関閾値 `max(0.15, 1.96/√n)` が n=35 で 0.331 になるため、大松丸 x アジ (deep_params 上位因子の |r|=0.443 があるのに combo_wx_params が空) の原因は **MIN_MONTHS ガードまたは cv_pct=NULL 問題** と推定される（後述 A-4）。

### A-4. factor 採用数の深刻な問題（0-factor コンボ）

chowari 専用 23件中 14件 が `combo_wx_params` 採用因子ゼロ。

```
0 factors:  14件 (61%) ← 気象補正が全く効かない「旬別平均」のみ予測
≥10 factors:  9件 (39%) ← 野毛屋釣船店 + 増福丸のみ
```

対して既存コンボの factor 採用数 median=21因子。

0-factor コンボの詳細分析 (大松丸 x アジ, n=35):
- deep_params は 19 件存在し、上位 6 因子は閾値 0.331 を超えている
- しかし `combo_backtest` に大松丸 x アジの行が**存在しない**（バックテスト自体が未実行）
- deep_params の n=19（全19行で n が統一）は MIN_MONTHS ガード（≥6か月）で walk-forward が走らなかった結果と推定
- chowari 専用船の釣行データが直近 1~3 か月に集中しており、月数を確保できていない

また `combo_meta` の全 0-factor コンボ (31件) を確認すると **cv_pct が全件 NULL** という別の問題も存在する。cv_pct は `combo_deep_dive.py` の `save_combo_meta()` で計算されるはずが、つね丸 x マダイ (avg_cnt=1.0, n=121) のように十分なデータがあっても NULL になっている。これはデータ品質問題の可能性がある（T33 で別途調査推奨）。

### A-5. factor の SLOW/FAST 偏り

chowari 専用コンボで factor が採用されているもの (増福丸・野毛屋釣船店) の因子分布:
- SLOW 因子: 185件
- FAST 因子: 4件 (SLOW に強く偏っている)

これは既存コンボと同傾向で、特異な偏りではない。

### A-6. 「新規コンボ特有の精度悪化パターン」の結論

- **wMAPE 悪化はない**（median ほぼ同一）
- **BL-2 勝率が 6.8pt 低い** = n 少ないため旬別平均に負けるケースが増加
- **最大の問題は 0-factor コンボが 61%**：気象補正が全く適用されず旬別平均のみの予測になっている
- 原因: MIN_MONTHS ガードで月数不足のコンボのバックテストが走らない → combo_wx_params に登録されない
- point_place1 取得の薄さは当初懸念の通り（chowari の kanso 記載が少ない）が、factor 採用に直接影響しているかどうかは combo_point_backtest では未確認

**改善アプローチ案:**
1. MIN_MONTHS を 6 から 4 に緩和（リスク: 過学習増加・適応的閾値でカバー可能）、難度: 低
2. データ蓄積待ち（自然蓄積で 3~6 か月後に解消）、難度: なし・ただし即効性なし
3. chowari 専用コンボは wMAPE 監視のみで現状維持・T34 以降に再評価、難度: なし

## D. その他の全体最適化候補

### D-1. DO 因子（溶存酸素: do_surface / do_bottom）の大規模採用漏れ

**現状:**
- 採用済み: 31コンボ (17魚種)、avg|r|=0.441 (採用因子中でも高水準)
- 未採用コンボ: 199件、うち採用閾値超えなのに未採用: **103件**
- 採用閾値超え未採用コンボの上位 |r|: 黒川本家-久里浜-×アジ (0.755)・栃木丸×マルイカ (0.737)・第八鶴丸×クロムツ (0.717)・博栄丸×カツオ (0.661)・棒面丸×マダイ (0.626)

**なぜ採用されていないか:**
- factor 採用数分布: median=23因子、最大=24因子で上位は `24 factors: 102コンボ`
- MAX_FACTORS=10 という設計だが **実際の combo_wx_params は 24 因子まで採用されている**（コード上で上限が機能していない可能性）
- DO 因子は採用閾値を大幅超えでも他因子に上書きされている理由が不明

**注記:** MAX_FACTORS=10 設計（設計ルール）と実績（最大24因子）の乖離を T33 着手前に確認推奨。もし上限が廃止されていれば「過学習対策」の前提が変わる。

**改善アプローチ:**
- DO 因子のみ強制採用オプション（MUST_FACTORS リスト化）、難度: 低
- または combo_deep_dive.py のステップデバッグで採用漏れ原因を特定・修正、難度: 中
- 期待効果: 採用 103 コンボで avg|r|=0.5 前後 → wMAPE 2~4pt 改善見込み
- リスク: DO データは cmems_data.sqlite に依存（GitHub Actions でアクセス不可）、予測時に NULL フォールバックが必要

### D-2. cooccurrence シグナルの予測変数化（潜在性中）

**現状:**
- cooccurrence テーブル: 807行、|r|>=0.40 の強相関ペア: 60件
- 代表ペア: マダコ×マルイカ (r=0.617)・ヒラメ×マルイカ (r=-0.584)・アジ×マダイ (r=0.577)

**潜在的活用方法:**
- 「同エリア当日の高相関魚種の釣果が既知の場合」、それを予測変数として当日予測に組み込む
- ただし予測用途（H≥1）では前日以前の値しか使えないため FAST 変数として H<=7 に制限が必要

**改善アプローチ:**
- prev_week_co_fish_N (前週同エリア高相関魚種の cnt_avg) を新 SLOW 因子として追加、難度: 中
- obs_fields.json に OBS 因子として先に定義し相関を取ってから判断するのが安全、難度: 低
- リスク: 同日 OBS 版はデータリーク（バックテスト外 → 実運用で欠損リスク）

### D-3. 0-factor コンボの根本原因調査（cv_pct=NULL 問題）

**現状:**
- 0-factor コンボ 31件のうち cv_pct が全件 NULL

- 例外: つね丸×マダイ (n=121, avg_cnt=1.0) → avg_cnt=1.0 固定値が続いているため相関が取れない可能性が高い
- ただし avg_cnt=43 の大松丸×アジも 0-factor → cv_pct=NULL と 0-factor は同じ原因ではない

**改善アプローチ:**
- `combo_deep_dive.py` の `save_combo_meta()` で cv_pct が NULL になるパスを確認、難度: 低
- 0-factor コンボのうち n>=50 かつ avg_cnt>=10 のコンボ（成銀丸×アジ, 大松丸×アジ, 黒川本家×アジ, 早川丸×イサキ等）を優先調査
- MIN_MONTHS ガードを 6→4 に緩和して再実行し 0-factor 解消数を計測、難度: 低

### 注意事項

1. **MAX_FACTORS=10 設計と実績 (最大 24 因子) の乖離を確認すること。** combo_wx_params の factor 数分布は `10: 1件, 11~15: 37件, 16~24: 318件` であり、設計上限が機能していない可能性が高い。過学習対策の前提として T33 着手前に combo_deep_dive.py の `_select_factors()` ロジックを確認推奨。
2. DO 因子は weather_cache.sqlite ではなく cmems_data.sqlite に依存。GitHub Actions 環境では cmems_data.sqlite がないため予測時 NULL フォールバックを必ず実装すること。
3. cooccurrence を予測因子化する場合、H=0 の OBS 用途と H>=1 の FAST 用途で分けて定義する必要がある。前者はバックテスト除外のため promise_break_rate への影響は測定不可。
