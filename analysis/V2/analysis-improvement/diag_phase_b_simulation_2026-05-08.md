## シミュレーション 1: pred_lo 下限比率

### データ

対象: combo_decadal（avg_cnt > 0 かつ avg_cnt_min not null）= **5,413 旬コンボ**

| floor 比率 | floor 発動行数 | 発動率 | avg_cnt_min/avg_cnt との関係 |
|---|---|---|---|
| 0.3 | 2,322 / 5,413 | **42.9%** | P50(0.337)をわずかに下回る水準 |
| 0.4 | 3,384 / 5,413 | 62.5% | P60 付近 |
| 0.5 | 4,395 / 5,413 | 81.2% | P71 付近 |
| 0.6 | 5,044 / 5,413 | 93.2% | P88 付近 |
| 0.7 | 5,314 / 5,413 | 98.2% | ほぼ全発動 |

### avg_cnt_min / avg_cnt の実分布

```
P10=0.121  P25=0.211  P50=0.337  P75=0.462  P90=0.561
```

### 評価

**floor=0.3 の適切性:**

- 発動率 43% は「過半数のコンボでは floor 不発動」を維持する。
- BL2 負けコンボ 148 件のうち avg_cnt_min/avg_cnt < 0.3 の件数は **96 件（64.9%）**。
  これらは pred_lo が floor によって引き上げられ、wmape 改善の期待が持てる。
- 残り 46 件（31.1%）は avg_ratio >= 0.3 のため floor 発動しない。
  これらは独立モデルがすでに 0.3*avg_cnt 以上を出力しており、floor の副作用もない。

**ゲーム化耐性:**

- floor=0.3 を設定しても、avg_cnt が小さいコンボ（例: avg_cnt=2 のコンボで floor=0.6 匹）では
  pred_lo が非現実的に高くなるリスクは低い。avg_cnt 自体が旬別値のため季節性を吸収している。
- 「pred_lo を 0 に設定して promise_break=0 を達成する」ゲーム化を防ぐには
  floor=0.3 が最低限必要であり、これより低い値は意味をなさない。

**過剰 floor の副作用（0.5 以上の問題）:**

- floor=0.5: 発動率 81%。ボウズ率が高いコンボ（キハダマグロ×ちがさき丸 ボウズ率 60.7% 等）
  では avg_cnt × 0.5 がボウズ日の actual=0 を上回り promise_break_rate が増加する。
- floor=0.7: 発動率 98%。実質的に「全コンボで pred_lo = 0.7*avg_cnt」に固定されて
  独立モデルの学習結果を完全に無視する設定になる。採用不可。

**推奨: floor = 0.3**

## シミュレーション 2: 区間幅上限倍率

### データ

対象: combo_decadal（avg_cnt > 0 かつ avg_cnt_min/max not null）= **5,413 旬コンボ**

平均幅比率（avg_cnt_max - avg_cnt_min）/ avg_cnt = **1.398**

| clamp 倍率 | 幅 > avg_cnt×倍率 の行数 | 発動率 | 評価 |
|---|---|---|---|
| 1.0 | 4,536 / 5,413 | **83.8%** | 過剰（P50 幅 1.387 を下回る） |
| 1.2 | 3,628 / 5,413 | 67.2% | 過剰（P50 超の大半で発動） |
| 1.5 | 2,208 / 5,413 | **40.8%** | 適度だが P75 超を全部カット |
| 2.0 | 291 / 5,413 | **5.4%** | 最小限介入（P90 超のみ） |
| 2.5 | 4 / 5,413 | 0.1% | 実質無制限 |

### 実データの幅分布

```
(cnt_max - cnt_min) / avg_cnt:
P10=0.908  P25=1.113  P50=1.387  P75=1.683  P90=1.903
```

### floor=0.3 との組み合わせシミュレーション（旬コンボ単位）

| floor | clamp | floor 発動 | clamp 発動 |
|---|---|---|---|
| 0.3 | 1.0 | 43% | 84% |
| 0.3 | 1.2 | 43% | 67% |
| 0.3 | 1.5 | 43% | **37%** |
| 0.3 | 2.0 | 43% | **1%** |

※ floor 適用後の pred_lo（= max(model_lo, 0.3*avg_cnt)）を起点に
  幅 = pred_hi - pred_lo > clamp*avg_cnt の場合に clamp 発動。

### 評価

**clamp=1.5 の問題:**

- 発動率 37% は「旬コンボの 37% で pred_hi を切り詰める」ことを意味する。
- P50 幅（1.387）が clamp=1.5 の境界に非常に近い。
  floor=0.3 適用後の pred_lo 引き上げにより実効幅がさらに縮まり、
  中央値コンボでも頻繁に clamp が発動する。
- over_expect_rate の高いコンボ（over > 40%、n=14、winkler=6.3）は幅が狭く
  coverage も 84.6% と高い。clamp=1.5 がこれらに当たると coverage が落ちる。

**clamp=2.0 の適切性:**

- 発動率 1%（291/5,413 旬コンボ）。P90 幅（1.903）超のみ介入。
- 「平均幅 2.0 倍超の非現実的レンジ」を持つコンボは全体の 5.4% にとどまり、
  これらに対してのみ clamp が効く設計。
- 発動率 1% は実質「上限外れ防止フェイルセーフ」として機能する。
  過剰に機能して coverage や promise_break を悪化させるリスクが最小。

**avg 幅比率 > 2.0 の実際のコンボ（clamp 頻繁発動コンボ）:**

旬単位で avg_width_ratio > 2.0 の件数が多いコンボ（clamp=2.0 で頻繁発動する）:

- キハダマグロ×幸栄丸（avg_width=2.170）
- カツオ×幸栄丸（avg_width=2.156）
- アオリイカ×工藤屋（avg_width=2.123）

これらは回遊魚・季節依存型コンボで実際に「爆釣日と坊主日の振れ幅が大きい」魚種。
clamp=2.0 でこれらのみ制約することは妥当。

**推奨: clamp = 2.0**

## シミュレーション 3: cnt_min BL-2 勝率改善見込み

### 現状（Phase A 再実行前・ratio 法適用時）

| metric | H=0 コンボ数(n≥30) | BL-2 勝率 | avg wmape | avg bl2_wmape |

| cnt_max | 317 | **92.1%** | 40.7% | 46.2% |

cnt_min の BL-2 勝率 51.7% は ratio 法を使っている現在のバックテスト値であることに注意。
BL-2 負けコンボの平均 gap = +1.5pt（wmape が bl2_wmape を +1.5pt 上回る）で微差。

### BL-2 負けコンボの構造分析

BL-2 負けコンボ 148 件の avg_cnt_min/avg_cnt 比率分布:

```
P10=0.145  P25=0.171  P50=0.236  P75=0.320  P90=0.388
```

BL-2 勝ちコンボ 166 件の avg_cnt_min/avg_cnt 比率分布:

```
P25=0.342  P50=0.411  P75=0.493
```

重要な発見: **BL-2 負けコンボは avg_cnt_min/avg_cnt 比率が有意に低い**（P50 で 0.236 対 0.411）。
これは「cnt_min が avg_cnt に対して非常に小さいコンボ」ほど BL-2 に負けていることを示す。

### floor=0.3 適用後の BL-2 勝率改善の理論的推計

BL-2 負け 148 件の内訳:

| 区分 | 件数 | 割合 | floor=0.3 の効果 |
|---|---|---|---|
| avg_cnt_min/avg_cnt < 0.3 | 96 件 | 64.9% | floor で pred_lo が引き上げられ、BL-2 との差が縮まる可能性あり |
| avg_cnt_min/avg_cnt >= 0.3 | 46 件 | 31.1% | floor 発動しない。独立モデルが既に 0.3*avg_cnt 以上を出力 |
| avg_ratio 不明 | 6 件 | 4.1% | combo_decadal 未登録（新規コンボ等） |

**理論的な BL-2 勝率の改善上限:**

floor=0.3 が 96 件全てに有効に機能した場合（楽観ケース）:
- 改善対象: 96 件（現状負け）が全て勝ちに転じると仮定
- 改善後勝ち件数: 166 + 96 = 262 件 / 314 件
- 改善後 BL-2 勝率: **83.4%**

実際には全件が勝ちに転じるわけではないため、現実的な期待値は **65〜75%** 程度。

## Plan #6 winkler 整合性についての判断

Plan で「保留」とされていた不明点 #6「防御策②の max_width = avg_cnt × 1.5 は winkler と整合するか」を解消する。

### winkler スコアの構造と clamp の影響

winkler スコア = 区間幅 + ペナルティ（区間外れの場合）

区間幅を clamp で狭めると:
- 区間幅ペナルティ: 減る（良化）
- 外れペナルティ: pred_hi が下がることで actual > pred_hi のケースが増加（悪化）

### 実データでの winkler 観察（cnt H=0、34 コンボ完了分）

| コンボ群 | over_expect | winkler 平均 | coverage | promise_break |
|---|---|---|---|---|
| over_expect > 40%（14 件） | 0.466 | **6.3** | 0.846 | 0.080 |
| over_expect <= 40%（32 件） | 0.264 | **16.0** | 0.809 | 0.109 |

over_expect が高い（pred_hi が実態より大きすぎる）コンボは **winkler が低い**。
これは「pred_hi が大きいと区間幅ペナルティが下がり、外れペナルティが減るので winkler が小さくなる」
というカウンターインテュイティブな結果。

### clamp=1.5 vs clamp=2.0 の winkler 影響

**clamp=1.5 の場合（発動率 37%）:**

- 37% のコンボで pred_hi が切り詰められる。
- これらは大部分が「over_expect が高いコンボ（winkler=6.3 の群）」に該当。
- clamp で pred_hi を切ると over_expect 改善（良化）の一方、
  actual > pred_hi となるケースが増加 → promise_break の悪化（外れペナルティ増加）。
- 2 つの効果が拮抗するため、winkler への純効果は不透明。
- **発動率 37% は過剰介入で、coverage/promise_break への副作用リスクが高い。**

**clamp=2.0 の場合（発動率 1%）:**

- 発動するのは P90 幅（1.903）を超える外れ値的コンボのみ（全体の 1%）。
- 「幅 2.0 倍超」のコンボは現実的に avg_cnt の 2 倍以上ばらついている魚種（回遊魚等）。
  これらは幅が広すぎて winkler ペナルティよりも over_expect の信頼性失墜が問題。
- clamp=2.0 での winkler への影響は **測定誤差範囲内（<0.1%）** と見込まれる。
- **winkler と整合的かつ promise_break への副作用が最小の設定。**

### Plan #6 解消結論

**防御策② clamp=2.0 は winkler と整合する。**

理由:
1. 発動率 1% により、winkler スコアへの影響は全体では誤差範囲内
2. 発動するコンボ（P90 超・幅 2.0 倍超）は over_expect_rate が高く、
   clamp によって winkler 悪化より over_expect 改善の効果が優勢
3. clamp=1.5 は発動率 37% で winkler/coverage への副作用が大きく採用不可
4. clamp=2.0 は「非現実的な幅を防ぐフェイルセーフ」として機能し、
   通常のコンボには影響しない

## 推奨採用値

```
防御策①: pred_lo >= avg_cnt × 0.3
防御策②: 区間幅 <= avg_cnt × 2.0
```

具体的な実装式（predict_count.py）:

```python
GUARD_LO_RATIO = 0.3    # 防御策①: pred_lo の下限
GUARD_WIDTH_MAX = 2.0   # 防御策②: 区間幅の上限倍率

# 防御策① 適用（cnt_lo 計算後）
floor_lo = avg_cnt * GUARD_LO_RATIO
cnt_lo = max(cnt_lo, floor_lo)

# 防御策② 適用（cnt_hi 計算後）
max_width = avg_cnt * GUARD_WIDTH_MAX
cnt_hi = min(cnt_hi, cnt_lo + max_width)
```
