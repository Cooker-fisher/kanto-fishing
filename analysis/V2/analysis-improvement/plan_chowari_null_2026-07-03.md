## 実装上の要注意（調査済み）

1. **fish ページ自動生成**: 新 tsuri_mono はデータに現れると `build_fish_pages()` が
   自動でページ化する（crawler.py:8374-8376）。`_FISH_ROMAJI` 未登録だと
   日本語ファイル名にフォールバック（crawler.py:3812）→ URL 規約違反。
   **昇格種は _FISH_ROMAJI へのスラッグ登録が必須**
2. **薄ページ/AdSense リスク**: fish/ は全ページ index 対象。昇格直後の新ページは
   hist が浅く AdSense 薄判定の温床になりうる → 昇格は高ボリューム種に絞る、
   または fish ページにも noindex 閾値を導入（fish_area #35 と同型）
3. **前方部分一致の誤爆**: normalize 3段目 `p in raw` により、短いパターンは
   別種を巻き込む（例: パターン「サバ」→「サバフグ」。ただし現状も「フグ」パターンが
   先に「サバフグ」を拾っており挿入順で既存キー優先のため悪化はしない）。
   検証スクリプトで全 raw 値の before/after 差分を必ず出す
4. **`raw in p` は禁止**（アマダイ→マダイ事故の前例。`p in raw` のみ）
5. **変更インパクト ★★★**: マップ変更後は chowari_to_csv.py 再実行（chowari分）+
   fishing-v 側 CSV は `FORCE_EXPORT=1 --export-csv` 全再生成 + C層フル再実行が本来セット。
   C層は次回定期実行に相乗りでも可（コンボ成立には n>=30 必要・当面は E層表示の充実が主目的）
6. **ALL_FISH への追加**: 昇格種を run_full_deepdive の ALL_FISH に足すのは
   n>=30 コンボが成立してからで良い（別判断）
