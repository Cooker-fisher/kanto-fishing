## 釣りドメイン知識（船宿船長由来）

### 基本方針

- **統計有意 ≠ 釣り的に有意** — 現場感と乖離した結果には異議を唱える
- **魚種・季節・潮汐の常識を適用** — 「その時期にその魚は釣れない」を指摘できる
- **変数選択の妥当性を助言** — 「SST より水温の変化量のほうが効く」等

### 変数ドメインチェック

| 変数・結果 | ドメイン的チェック |
|-----------|-----------------|
| SST | アジ適水温 16〜19℃。20℃超えで釣果低下は妥当 |
| 月齢 | タチウオは大潮に強い。カワハギは影響薄 |
| 潮汐 | 潮が動かないと魚は食わない（全般） |
| 南西風 | 荒れやすい。出船リスク要注意 |
| 乗っ込み | マダイ(4〜6月)・シーバス(2〜4月)・サワラ(3〜5月東京湾接岸) |
| 潮流 | 潮止まり近い時間帯は小型が主体・バラシ多数 |
| spawn_season_n | 2〜5月=1（乗っ込みシーズン）SLOW因子として全H有効 |

### 回遊魚ドメイン知識

| 魚種 | 接岸時期 | 釣果特性 |
|------|---------|---------|
| カツオ | 7〜9月 | 群れの有無が全て。MAPEより★チャンス評価が向く |
| キハダ | 8〜10月 | カツオと同様。kaiyu_promoted後は匹数レンジ |
| サワラ | 3〜5月（接岸）・9〜12月 | 潮流依存が強い |
| ブリ・ワラサ・イナダ | 10〜1月 | 水温低下で接岸 |
| シイラ | 7〜9月 | 外洋性。沖合SST依存 |

## E層表示設計（下層担当由来）

### 基本方針

- **「釣り人に伝わるか」が最優先** — MAPE 35% より ★4 の方が直感的
- **分析結果 → 表示形式の変換を設計** — ★評価・平年比±%・理由テキストの要件を定義
- **有料/無料の区分を守る** — 予測コメントは有料、数字は無料
  - 根拠: `design/V2/90_決定ログ.md` が正

### 表示設計の責務

| 出力形式 | 説明 | 有料/無料 |
|---------|------|---------|
| ★1〜5評価 | 旬別平年比に基づく直感的評価 | 無料（★のみ） |
| cnt_lo〜cnt_hi | 匹数レンジ予測 | 有料 |
| 平年比 ±% | 旬別ベースライン比 | 有料 |
| 理由テキスト | 「SST高め・潮汐好条件」等 | 有料 |
| ★チャンス評価 | 回遊魚の来訪確率 | 有料 |

### C層→E層データフロー設計

```
analysis.sqlite (combo_decadal, combo_backtest, combo_wx_params)
        ↓ predict_count.py が読む
        ↓
予測結果 JSON (crawler.py が呼ぶ)
        ↓
forecast/index.html / fish/*.html
```

E層の表示要件変更があれば design/V2 チームと調整すること。独自にE層設計を決めない。

## obs_fields.json 管理

新しいキーワードスコアを追加する場合は `normalize/obs_fields.json` を更新:

```jsonc
"tide_speed_n": {
  "source": "kanso_raw",
  "compute": "keyword_score",
  "map": {
    "潮止まり": -1.0,
    "潮止り": -1.0,
    "止まり": -0.5
  },
  "role": "obs_factor",
  "desc": "潮流スコア（観測値）"
}
```

キーワードの追加根拠は必ずドメイン知識（現場の声・kanso_rawの実例）から取ること。
