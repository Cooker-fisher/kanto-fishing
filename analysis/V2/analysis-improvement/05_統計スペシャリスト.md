## MAPE改善の方向性（初期仮説）

1. **外れ値処理** — 極端な釣果データ（0匹・100匹超等）のクリッピングまたは除外
2. **変数選択の最適化** — 魚種×船宿ごとに有効な変数セットが異なる可能性
3. **ベースラインの精緻化** — combo_decadal（旬別平均）を月別・半旬別に細分化
4. **非線形効果の考慮** — 潮汐・月齢の閾値効果（大潮のみ有意等）
5. **欠航除外の徹底** — 欠航日データが平均を引き下げていないか

---

## 品質ルール

1. **改善効果はバックテスト（backtest_oos.py）で定量評価** — 「良くなった気がする」禁止
2. **変数の有効ホライズンを超えた使用は禁止** — D2でwave_heightを使わない
3. **過学習チェック** — インサンプルMAPE vs OOS MAPE の差が10%以上なら要見直し
4. **信頼区間を出す** — 点推定だけで判断しない

## ⚠️ バックテスト vs 予測の設計原則（2026/04/09 確定）

このプロジェクトには**2つの異なる精度検証フェーズ**がある。

### フェーズ1: バックテスト最適化（現行）
- **入力**: 気象**実績値**（actual wave height, actual SST 等）
- **目的**: 変数選択・相関検証・モデル構造の最適化
- **注意**: 実績値で最適化したパラメータが予測時にも最適とは限らない

### フェーズ2: 予測精度テスト（今後実装）
- **入力**: 気象**予報値**（Open-Meteo forecast API 等、誤差を含む）
- **目的**: 実運用での予測精度を検証
- **注意**: 予報誤差が大きい変数（波高・風速）ほど実績値との乖離が大きい

### 設計上の原則
| 変数種別 | 実績値との差 | バックテストでの扱い |
|---------|------------|------------------|
| is_holiday / 潮汐 / 月齢 | 誤差ゼロ（確定値） | そのまま使ってよい |
| SST / 気温 / 気圧 | 誤差小（週単位で安定） | バックテストで有効 |
| 波高 / 風速 | 誤差大（数日で急変） | H>7では無効化済み。予報誤差も要考慮 |

**コンボ別パラメータ最適化（wave_clamp等）は、予報値ベースの精度テストで検証してから本採用すること。**

### フェーズ2 評価設計（2026/04/09）

#### 評価プロトコル
- **評価単位**: コンボ（魚種×船宿）× H=7日前予測
- **最低サンプル**: n≥30件（週1釣行なら約7ヶ月。先行検証はアジ×東京湾奥から）
- **集計方式**: ローリング（毎日蓄積）。月次ではなく日次更新

#### 劣化想定（フェーズ1 wMAPEからの上乗せ）
| 変数 | 劣化想定 | 理由 |
|------|---------|------|
| 潮汐 / 月齢 / is_holiday | 0pt | 確定値、誤差なし |
| SST / 気温 / 気圧 | +2〜10pt | 週単位変化で予報精度高い |
| 波高 / 風速 (H=7) | +15〜25pt | 天気予報誤差が直撃 |
| 全体 | **+10〜20pt** | フェーズ1中央値44.8% → 55〜65%想定 |

#### 追加評価指標
- **Skill Score**: `1 - (RMSE_forecast / RMSE_climatology)` — 旬別ベースラインに勝てているか？
- **Forecast Error Correlation (FEC)**: 予報誤差と釣果予測誤差の相関 — 「波高予報が外れたから外れた」を定量化
- Skill Score < 0 の変数・ホライズンはカット候補

#### prediction_log テーブル設計
```sql
CREATE TABLE prediction_log (
    fish TEXT, ship TEXT,
    target_date TEXT,    -- 釣行予定日
    pred_date TEXT,      -- 予測実行日
    horizon INTEGER,     -- target_date - pred_date
    pred_cnt_avg REAL, pred_cnt_min REAL, pred_cnt_max REAL,
    pred_stars INTEGER,
    fcast_wave REAL, fcast_wind REAL, fcast_sst REAL, fcast_temp REAL,
    actual_cnt_avg REAL, actual_cnt_min REAL, actual_cnt_max REAL,
    actual_wave REAL, actual_wind REAL,
    wmape REAL, mae REAL, is_good_hit INTEGER,
    created_at TEXT, matched_at TEXT
);
```

#### 実装ステップ（3段階）
1. `prediction_log.py` — テーブル初期化・スキーマ作成
2. `daily_predict()` — today+7 の全コンボ予測を INSERT（まず dry-run で件数確認）
3. `match_actuals()` — target_date <= today-1 の行に actual_* を UPDATE・精度計算

## テーマ4: 答え合わせ分析方法（主担当）

### 検討事項
- 予測結果の事後検証フレームワーク → **フェーズ2設計に統合**
- 日次バッチでの自動答え合わせ → prediction_log の match_actuals() で実現
- MAPE以外の評価指標（MAE, RMSE, Skill Score, FEC）→ 上記に追加済み
- 魚種×船宿単位での精度トラッキング → prediction_log テーブルで実現
- 「予測が外れた理由」の自動分類 → FEC（予報誤差相関）で原因帰属
