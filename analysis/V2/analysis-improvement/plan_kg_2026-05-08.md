## 1. 背景・目的

### 1.1 現状の問題

`predict_count.py` の kg_lo/hi は旧設計（avg ± mae）:

```python
# predict_count.py L1394-1395（現状・変更対象）
"kg_lo": round(max(0.0, avg_kg - kg_mae), 2) if avg_kg and kg_mae else None,
"kg_hi": round(avg_kg + kg_mae, 2)           if avg_kg and kg_mae else None,
```

この設計は「kg_avg の予測 MAE ≈ 1.0kg 程度」をレンジ幅として使うが、実測では
kg_min/avg の P20 = 0.300、kg_max/avg の P80 = 1.700 と、実測幅が MAE の 5 倍以上広い。
結果: kg promise_break P50 = 55.56%（H=0, n>=30, 103 コンボ）。

### 1.2 Phase B-α' との対比

size（Phase B-α'）と kg の主な違い:

| 観点 | size | kg |
|---|---|---|
| 旧 promise_break P50 | 98.4%（同一定義）| 55.56% |
| 中央予測 wMAPE P50 | 7.3% | 33.15% |
| P20 比率（min/avg）| 0.612 | 0.300 |
| P80 比率（max/avg）| 1.392 | 1.700 |
| 旬別 n>=10 の割合 | 81.4% | 32.5%（MIN_N=10）/ 60.5%（MIN_N=5）|
| 「お化け」（外れ値）| あり（P95 除外が有効）| なし（kg_max ≤ 2×avg は数学的必然）|
| 非対称設計 | あり（size_max 許容）| なし（対称評価でよい）|

### 1.3 Phase B-β-4 の名称

kg 専用の Phase B-β-4（Phase B-β = cnt 系の続き・4 = kg 先行番号）。
cnt は promise_break P50 = 6.52% で既に良好なため cnt 優先度は低く、kg を先行する。

## 2. ユーザー判断4点の確定値と Plan 反映

### 判断 1: kg 表示の優先度 → 数/型/重 3レンジ並列表示

**確定値**: cnt/size/kg を同等扱いで並列表示する。補遺3「3 種すべて min〜max のみ」と整合。

**実装影響**:

### 判断 2: NULL 71% コンボの扱い → kg 記録があるコンボのみ表示

**確定値**: kg_avg が NULL のレコードは重レンジを非表示にする(NULL ≠ 0)。

**実装影響**:
- predict_combo の返り値で `avg_kg is None` の場合は `kg_lo = None`, `kg_hi = None`
- これは現状も同様の動作(`if avg_kg and kg_mae else None`)を維持するため追加実装不要
- combo_decadal の kg 集計では kg_min/kg_max を NULL 除外後に集計する(save_decadal 既存設計)
- programmer への指示: kg_avg が旬別で 0 件(NULL)の旬はフォールバックも適用しない
- **data-reviewer 懸念への対応**: `avg_kg = 0` はゼロガードで `None` 扱い(kg が 0kg はデータ異常)。1コンボ動作確認で `avg_kg = 0` と `None` の混在がないか §5 で検証する

### 判断 3: キハダマグロ等の例外コンボ → 全コンボで Phase B-α' 同型適用

**確定値**: 個別 use_fallback 設定なし・シンプル実装。
キハダマグロ(pb=61.6%/cov=0.25/winkler=21.0)でも魚種別 P20/P80 を適用する。

**実装影響**:
- `use_fallback` テーブルに kg 用エントリを追加しない
- Phase B-α' と同型のコードパス(全コンボ一律)を採用
- キハダマグロの悪化懸念は撤回基準で判定(15pt 改善を見ないなら全体撤回でシンプル)

### 判断 4: 魚種別 P20/P80 → 魚種別に保持

**確定値**: 全期間グローバル一律(P20=0.300/P80=1.700)ではなく、魚種別に保持する。
理由: マダイ P80=1.739 / キハダマグロ P80=1.358 など魚種間で P80 に 0.4 の差がある。
size では全コンボ横断が有効だったが、kg は魚種間差が大きいため魚種別が適切。

**実装影響**:
- `combo_decadal` に `avg_kg_min` / `avg_kg_max` を列追加(旬別比率)
- 集計は「旬別 P20/P80」を base に、n<MIN_N_DECADAL_KG=5 で「魚種別グローバル比率」にフォールバック
- 魚種別グローバル比率は `combo_decadal` 全期間・全旬から事前集計して `_KG_GLOBAL_RATIO` 定数化

### 3.1 combo_deep_dive.py の save_decadal 拡張

#### テーブル構造(列追加方式)

`combo_decadal` に 2 列を追加する(size と同型の ALTER TABLE 方式):

#### 3.4.1 `_bt_decadal_kg_ratio` 構築(deep_dive 関数内)

size の `_bt_decadal_size_ratio` 構築(L2925-2982)に対応する kg 版を、`_bt_decadal_size_ratio` 定義ブロック(L2982 終端)の直後に追加する。

```python
# ── Phase B-β-4: バックテスト評価用 旬別 kg 実測比率ルックアップテーブル ──────────
# 全期間 records（H=0 enrich 済み）から旬別 kg_min/kg_avg 比率・kg_max/kg_avg 比率を計算。
#
# 【設計方針】(2026-05-08 補遺9 確定予定)
#   比率 P20/P80 を pred_lo/pred_hi に使用:
#     ratio_min: 旬別 kg_min/kg_avg の P20 → pred_lo = pred_avg × ratio_min
#     ratio_max: 旬別 kg_max/kg_avg の P80 → pred_hi = pred_avg × ratio_max
#   size と同型・P95 除外なし（kg_max ≤ 2×avg は数学的必然）
#
# 【MIN_N 設計】
#   _MIN_N_BT_KG = 5（size の 10 より緩い・kg は旬別データが薄いため）
#   n < 5 の旬はグローバル比率にフォールバック
#
_MIN_N_BT_KG = 5
_bt_kg_ratio_buckets = {}
for _r in all_en_by_H.get(0, []):
    _dec = _r.get("decade")
    if _dec is None:
        continue
    _km = _r.get("kg_min"); _kx = _r.get("kg_max")
    _ka = _r.get("kg_avg")
    # 妥当性チェック: kg_min < kg_avg < kg_max を満たす行のみ（size と同型）
    # ゼロガード: _ka <= 0 は除外
    if _km is None or _kx is None or not _ka or _ka <= 0 or not (_km < _ka < _kx):
        continue
    if _dec not in _bt_kg_ratio_buckets:
        _bt_kg_ratio_buckets[_dec] = {"ratio_min": [], "ratio_max": []}
    _bt_kg_ratio_buckets[_dec]["ratio_min"].append(_km / _ka)
    _bt_kg_ratio_buckets[_dec]["ratio_max"].append(_kx / _ka)

# グローバル（全旬）フォールバック
_all_kg_ratio_min_vals = [v for d in _bt_kg_ratio_buckets.values() for v in d["ratio_min"]]
_all_kg_ratio_max_vals = [v for d in _bt_kg_ratio_buckets.values() for v in d["ratio_max"]]
_global_kg_ratio_min_p20 = _bt_percentile(_all_kg_ratio_min_vals, 0.20) or 0.30
_global_kg_ratio_max_p80 = _bt_percentile(_all_kg_ratio_max_vals, 0.80) or 1.70

# 旬別比率テーブル
_bt_decadal_kg_ratio = {}
for _dec, _d in _bt_kg_ratio_buckets.items():
    if len(_d["ratio_min"]) >= _MIN_N_BT_KG:
        _rmin = _bt_percentile(_d["ratio_min"], 0.20)
        _rmax = _bt_percentile(_d["ratio_max"], 0.80)
    else:
        _rmin = _global_kg_ratio_min_p20
        _rmax = _global_kg_ratio_max_p80
    # 物理制約: ratio_min ∈ (0, 1]、ratio_max ∈ [1, ∞)
    _bt_decadal_kg_ratio[_dec] = (
        max(0.0, min(1.0, _rmin)) if _rmin is not None else _global_kg_ratio_min_p20,
        max(1.0, _rmax)           if _rmax is not None else _global_kg_ratio_max_p80,
    )
```

#### 3.4.2 kg 評価ループの比率ベース化(L3677-3714 を全面置換)

現状のkg評価ループ(L3677-3714)は「点予測 pred ± 実測レンジ」の旧 Phase A 設計:

```python
# 現状（旧設計・Phase A・L3677 冒頭コメント）
# kg は NULL 率 70.2% でサンプル少のため Phase B-α' 拡張対象外。
# promise_break = actual_kg_avg < pred（点予測を下回った）
```

この旧設計ブロック全体(L3677-3714)を以下の比率ベース設計に置換する:

```python
# ── kg メトリック（Phase B-β-4: 実測幅ベース比率評価）──────────────────────────
# size（Phase B-α'）と同型。点予測 pred × 旬別 P20/P80 比率でレンジを生成。
#   promise_break (主指標) = actual_kg_min < pred_lo（下限割れ）
#   over_expect   (副指標) = actual_kg_max > pred_hi（上限超過）
#     → kg は size と異なり非対称設計なし。over_expect も記録して KPI 参考とする
#   coverage               = actual_kg_min >= pred_lo かつ actual_kg_max <= pred_hi
#   winkler                = promise 側（actual_kg_min）で評価
#
# 分母統一: actual_kg_min / actual_kg_max 両方非 NULL の行のみ（n_valid ベース）
for H in HORIZONS:
    rows_kg = []
    for _rk, d in _range_by_key[H].items():
        if "kg_avg" not in d:
            continue
        pred_c, act_avg = d["kg_avg"]
        if act_avg is None or pred_c is None:
            continue
        act_min = d.get("_kg_min_act")
        act_max = d.get("_kg_max_act")
        dec_no  = d.get("_decade_no")
        # 旬別比率ルックアップ（フォールバック: グローバル P20/P80）
        _rmin, _rmax = _bt_decadal_kg_ratio.get(
            dec_no, (_global_kg_ratio_min_p20, _global_kg_ratio_max_p80)
        )
        pred_lo = pred_c * _rmin
        pred_hi = pred_c * _rmax
        rows_kg.append((pred_lo, pred_hi, pred_c, act_avg, act_min, act_max))

    rows_valid = [(pl, ph, pc, aa, ami, amx) for pl, ph, pc, aa, ami, amx in rows_kg
                  if ami is not None and amx is not None]
    n_valid = len(rows_valid)
    if n_valid < MIN_N_RANGE:
        continue

    promise_break_rate = sum(1 for pl, ph, pc, aa, ami, amx in rows_valid
                             if ami < pl) / n_valid
    over_expect_rate   = sum(1 for pl, ph, pc, aa, ami, amx in rows_valid
                             if amx > ph) / n_valid
    coverage           = sum(1 for pl, ph, pc, aa, ami, amx in rows_valid
                             if ami >= pl and amx <= ph) / n_valid
    bowzu_rate         = None  # kg はボウズ概念なし
    winkler            = sum(_winkler_range(pl, ph, ami)   # promise 側は min で評価
                             for pl, ph, pc, aa, ami, amx in rows_valid) / n_valid

    lh = "H=  0(実測)" if H == 0 else f"H={H:>3}d 前"
    lines.append(
        f"  {'kg':>6}  {lh:>12}  {promise_break_rate:>8.1%}  "
        f"{over_expect_rate:>8.1%}  {coverage:>7.1%}  {'N/A':>6}  "
        f"{winkler:>7.2f}  {n_valid:>4}"
    )
    range_bt_data.append(("kg", H, promise_break_rate, over_expect_rate,
                           coverage, bowzu_rate, winkler, n_valid))
```

**combo_range_backtest への書き込み**: `range_bt_data` の kg 行が新定義(比率ベース)で生成され、`combo_range_backtest` の `metric='kg'` 行が更新される。これにより撤回基準 SQL(§9 Phase 6)が正しく動作する。

## 4. 撤回基準（1つに絞る）

**撤回基準: kg promise_break 改善 ≥ 15pt（H=0, n>=30, P50 が 55.56% → 40.56% 以下）**

### 根拠

- 現状 P50 = 55.56%
- 期待改善幅: 10〜20pt（diag §6.3）
- 中央予測 wMAPE P50 = 33.15%。pred_avg が正確であれば改善 ≈ 20pt+。wMAPE 考慮後の下限 10pt と上限 20pt の中央付近
- 15pt 基準は「10-20pt 期待の中央値」。誤差レベル(10pt)より上・size(30pt) の半分

### 補遺8 の学び「撤回基準二重定義は危険」の適用

- plan_size には「P50=23%」という暗黙の到達目標があり、同一定義 30pt 改善と二重定義になった
- 本 Plan では **「改善幅 15pt 以上」1 基準のみ**(絶対値目標は設けない)
- 同一定義確認: 旧 kg_lo(avg_kg - kg_mae)でも新定義(actual_kg_min < pred_lo)で計算してから比較する

### MIN_N=5 の統計的留保（stat-reviewer MAJOR 1 への対応）

**v1からの追加**: stat-reviewer 指摘「diag §3.3 の P80 CV=2% は n>=10 の 36 旬の旬間ばらつきであり、n=5 での個別パーセンタイル推定精度とは別概念」を §4 リスク説明に明示する。

Beta 分布評価:
- n=5 で P80 推定誤差 ≈ ±16%（n=10 の ±12% の 2倍以上）
- 「不安定推定が偶然グローバル 0.300 に近いから改善して見えた」という解釈が排除できない

**対応フロー(MIN_N=5 vs 10 の最終判断を動的化)**:
1. §5 1コンボ動作確認で MIN_N=5 と MIN_N=10 両方走らせて NULL 率と avg_kg_min/max の分布を比較
2. NULL 率の差が大きい場合(MIN_N=10 で旬別充足率 < 30% 等)は MIN_N=5 採用
3. NULL 率の差が小さい場合は MIN_N=10 への引き上げを pm が判断
4. 最終決定後に全コンボ再実行(MIN_N が実装定数に反映されていることを確認してから)

### 撤回しない副作用

size と同様に以下は「数学的必然」として許容:
- coverage の低下(P20/P80 の理論 coverage = 60%。現状 67.9% から低下は自然)
- over_expect の増加(kg は対称評価なので over_expect も記録するが KPI 外)
- winkler の増加(区間幅が広がるためペナルティ増は必然)
