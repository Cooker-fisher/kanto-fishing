#### 2-B. `_apply_factor_caps` 修正（魚種別 DO 制御を追加）

シグネチャに `fish` を追加し、`DO_INEFFECTIVE_FISH` に該当する魚種は DO 採用を完全スキップ:

```python
def _apply_factor_caps(factor_r_dict: dict, fish: str = "",
                       max_total: int = MAX_FACTORS,
                       max_cmems: int = MAX_CMEMS_DEFAULT,
                       max_do: int = MAX_DO,
                       corr_thr: float = 0.10) -> dict:
    """v2 改訂: fish 引数で DO の魚種別制御を追加"""
    # DO 逆効果魚種は事前除去
    if fish in DO_INEFFECTIVE_FISH:
        factor_r_dict = {k: v for k, v in factor_r_dict.items() if k not in DO_FACTORS}

    base_selected = {f: r for f, r in factor_r_dict.items() if f in BASE_FACTORS}
    tier2_cands   = {f: r for f, r in factor_r_dict.items() if f not in BASE_FACTORS}

    tier2_sorted = sorted(tier2_cands.items(), key=lambda x: abs(x[1]), reverse=True)
    tier2_selected: dict = {}
    cmems_cnt = 0
    do_cnt = 0
    for fac, rv in tier2_sorted:
        if fac in DO_FACTORS:
            if do_cnt >= max_do:
                continue
            do_cnt += 1
        elif fac in CMEMS_FACTORS:
            if cmems_cnt >= max_cmems:
                continue
            cmems_cnt += 1
        tier2_selected[fac] = rv
        if len(tier2_selected) >= MAX_TIER2:
            break

    return {**base_selected, **tier2_selected}
```

### Phase 4: predict_count.py の DO=NULL フォールバック実装（v2 新規）

code-reviewer M-1 への対処。

`predict_count.py` の `_build_all_wx()` に DO 取得経路がないため、本番予測（GitHub Actions・cmems_data.sqlite なし）で **DO 値が常に NULL** になる。バックテスト（ローカル・DO あり）と本番予測のギャップが系統的バイアスを生む。

#### 4-A. 実装案（最小変更）

```python
# predict_count.py の _build_all_wx() 内
def _build_all_wx(date_str, lat, lon):
    ...
    # DO（cmems_data.sqlite 参照・ローカルのみ）
    do_surface = None
    do_bottom = None
    try:
        if os.path.exists(CMEMS_DB):
            con = sqlite3.connect(CMEMS_DB)
            cur = con.cursor()
            # 最近傍座標・最近傍日付の DO 取得（深度 0.5m と最深）
            cur.execute("""
                SELECT temp, do FROM cmems_depth 
                WHERE date <= ? AND lat BETWEEN ? AND ? AND lon BETWEEN ? AND ?
                ORDER BY date DESC, depth_m ASC LIMIT 1
            """, (date_str, lat-0.25, lat+0.25, lon-0.25, lon+0.25))
            row = cur.fetchone()
            if row:
                do_surface = row[1]  # 最浅 DO
            cur.execute("""
                SELECT do FROM cmems_depth 
                WHERE date <= ? AND lat BETWEEN ? AND ? AND lon BETWEEN ? AND ?
                ORDER BY date DESC, depth_m DESC LIMIT 1
            """, (date_str, lat-0.25, lat+0.25, lon-0.25, lon+0.25))
            row = cur.fetchone()
            if row:
                do_bottom = row[0]  # 最深 DO
            con.close()
    except Exception as e:
        pass  # DB 不在時はサイレント NULL
    return {
        ...,
        "do_surface": do_surface,
        "do_bottom": do_bottom,
    }
```

#### 4-B. 設計の割り切り（ユーザー承認事項）

GitHub Actions 環境では cmems_data.sqlite が存在しないため、DO 採用コンボの本番予測は **DO=NULL のまま** で動作する。これは既存の他 CMEMS 因子（sla/chl/temp_100m）も同様で、`_apply_correction()` の NULL ハンドリングで「該当因子をスキップ」が既に実装されている。

→ DO 採用拡大による本番予測精度の改善は **ローカル予測のみで実現**。GitHub Actions では依然として他因子のみで予測。

#### 4-C. ユーザー判断要件

cmems_data.sqlite を GitHub Actions リポジトリに含めるかは別議論（容量・更新頻度・ライセンス）。T33 ではローカルのみで動作する設計を維持。
