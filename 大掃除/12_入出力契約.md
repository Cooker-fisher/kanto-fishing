## バージョン管理アーキテクチャ（2026/04/08 確定）

### 設計方針
分析条件・分析項目が変わるたびにバージョンを上げる。
**全コンポーネントが `config.json` の `active_version` を参照し、同じバージョンで会話する。**

### ファイル構成

```
config.json                  ← ★唯一の真実。全員がここを見る
analysis/
├── analysis_config.py       ← 後工程（crawler.py等）が使うユーティリティ
├── run.py                   ← crawl.yml のランチャー（バージョン自動解決）
├── V1/
│   ├── README.md            ← V1の条件・前後工程との契約
│   ├── methods/             ← 旧スクリプト群
│   └── results/             ← 旧出力物
└── V2/（現行）
    ├── README.md            ← V2の条件・前後工程との契約
    ├── methods/
    │   └── _paths.py        ← スクリプト内のパス自動解決
    └── results/             ← analysis.sqlite, deep_dive/ 等
```

### config.json（プロジェクトルート）

```json
{
  "active_version": "V2",
  "versions": {
    "V1": {"period": "〜2026-03", "note": "旧フォーマット"},
    "V2": {"period": "2026-04〜", "note": "weather_cache.sqlite導入・51魚種"}
  }
}
```

### 各コンポーネントの役割

| ファイル | 役割 | バージョン切替時 |
|---------|------|----------------|
| `config.json` | active_version の一元管理 | **ここだけ変える** |
| `analysis/run.py` | crawl.yml → active版 methods/ を呼ぶ | 変更不要 |
| `analysis/analysis_config.py` | crawler.py → active版 results/ を参照 | 変更不要 |
| `analysis/Vn/methods/_paths.py` | スクリプト内の ROOT_DIR / RESULTS_DIR 解決 | Vn+1にコピーするだけ |
| 各フォルダの `README.md` | バージョン別の入出力契約を記述 | 変更があれば更新 |

### バージョンアップ時の手順

```
1. config.json の active_version を "V3" に変更
   → crawl.yml・crawler.py が自動でV3を参照
2. analysis/V3/methods/ を作成
3. _paths.py を V2 からコピー（中身変更不要）
4. 新しいスクリプトを書く（from _paths import ... の1行だけ追加）
5. analysis/V3/README.md に「V2からの変更点・前後工程との契約」を記述
6. 変更があるコンポーネントのREADMEのバージョン契約欄を更新
```

## 契約一覧

### crawl/ → normalize/

| 出力ファイル | 形式 | 内容 | 更新頻度 |
|-------------|------|------|---------|
| catches_raw.json | JSON配列 | 釣果生データ（84,757件〜） | 毎日自動 |
| catches.json | JSON配列 | 当日スナップショット | 毎日自動 |
| catches_all.json | JSON配列 | 全釣果（差分追記）crawler.py が直接読み書き | 毎日自動 |

> ⚠️ catches_all.json は crawler.py 内で①差分追記②TOP船宿表示③7日間グラフ生成の3箇所で参照。Phase 2 のパス修正では最多参照ファイルになるため、GitHub担当者による事前grep洗い出しが特に重要。

## 契約変更ルール

1. 出力フォーマット（JSON/CSVのカラム）を変更する場合、**利用先フォルダの担当に通知必須**
2. ファイル名変更は**全利用先のパス修正とセット**（GitHub担当者に依頼）
3. 新しい入出力が発生したら、この文書を更新してからコードを変更する
