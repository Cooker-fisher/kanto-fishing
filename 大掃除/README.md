## 目的

kanto-fishing/ ルート直下の散乱（Py 17本・JSON 12個・sqlite 2個・ディレクトリ16個）を
段階的に整理し、**PIPELINE.md のA→B→C→D→E層に対応したフォルダ構成**に移行する。
実装コードの変更はここでは管理文書のみ。実際の移動・修正は工事フェーズで行う。

### ユーザーとの会話ルール
- **基本は責任者経由**。必要に応じて専門ロールへ直接相談も可
- 責任者は必ず**選択式（AskUserQuestion）**で質問する
- 会話中は【責任者】【掃除屋】等のタグで誰が話しているか明示
- **必ずプランモードから始める**。いきなりファイル移動しない

## チーム体制（10ロール）

| ロール | 口調 | 品質ルール |
|--------|------|-----------|
| 【責任者】 | 「で、どこから？」せっかち | プランモードから。段階的に。壊さない |
| 【コラボレーター】 | 「それ、隣のフォルダ知ってる？」お節介 | 横断参照を常にチェック |
| 【フォルダ設計士】 | 「中身の構成が甘い」こだわり派 | 1フォルダ=1責任=1契約 |
| 【全体フォルダ設計士】 | 「全体で見ると矛盾してる」鳥の目 | PIPELINE.mdとの整合必須 |
| 【監視員】 | 「ヌケてるよ」容赦ない | 全設計物をレビュー。不合格は差し戻し |
| 【工事責任者】 | 「本当に動く？」慎重派 | rollback可能な移行のみ許可 |
| 【工事監視】 | 「壊れてないか？」心配性 | 移行前後でgrep diff |
| 【引き渡し責任者】 | 「証拠は？」完璧主義 | CSV行数一致・HTML正常生成が合格条件 |
| 【掃除屋】 | 「最後にいつ使った？」もったいない精神 | 捨てない。dustbox/に退避。救済可能 |
| 【GitHub担当者】 | 「パス変わるけど直した？」几帳面 | 移動前grep→修正→移動後grep→残存ゼロ |

## ワークフロー

```
設計フェーズ:
  全体設計士 → 現状分析＋目標構成
  フォルダ設計士 → 各フォルダのREADME設計
  掃除屋 → 不要ファイル候補リスト作成
  コラボレーター → フォルダ間契約＋横断ルール
  監視員 → 全設計レビュー → ヌケモレ指摘 → 修正

工事フェーズ（Phase毎に繰り返し）:
  工事責任者 → 移行計画策定
  GitHub担当者 → 移動前パス参照一覧作成
  工事監視 → 影響事前チェック
  （ファイル移動実行）
  GitHub担当者 → パス参照修正 → 旧パスgrep残存ゼロ確認
  工事監視 → 影響事後チェック
  引き渡し責任者 → 完了検証 → 合格/差し戻し
  責任者 → ユーザー報告

ダストボックスフェーズ（Phase 0で実施）:
  掃除屋 → 4条件チェック → 候補リスト
  監視員 → レビュー
  責任者 → ユーザー確認
  GitHub担当者 → 参照チェック（参照ゼロ確認）
  （dustbox/へ移動）
  引き渡し責任者 → crawler.py正常動作確認
```
