# 05_実装方針 — パーサー設計・統合方針

> **変更ガバナンス**: catches_raw.json / data/*.csv のフィールド変更はオーナー最終承認 + 関連MD責任者了解が必須。

## 基本方針

- **新スクリプト**: `direct_crawler.py`（crawler.py とは完全分離）
- **既存パイプラインを壊さない**: catches_raw.json の既存フォーマットに合わせてマージ
- **個別失敗で全体を止めない**: 船宿ごとに try/except、失敗は skip して続行

---

## ディレクトリ構成

```
kanto-fishing/

└── ...
```

## ships.json への追加フィールド

```json
{
  "sid": 12345,
  "name": "忠彦丸",
  "area": "金沢漁港",
  "homepage": "https://tadahikomaru.jp",
  "catch_page": "https://tadahikomaru.jp/chouka/",
  "parser_type": "type_e"
}
```

| フィールド | 型 | 説明 |
|-----------|-----|------|
| homepage | string / null | 船宿トップページURL |
| catch_page | string / null | 釣果ページURL（直接パースする対象） |
| parser_type | string / null | パーサータイプ（"type_a" / "type_b" / "type_c" / "type_e" / null） |

- `parser_type` が null の船宿はスキップ（釣りビジョン経由のみ）
- `catch_page` が null の船宿もスキップ

---

## direct_crawler.py の処理フロー

```
1. ships.json を読み込む
2. parser_type が設定されている船宿をフィルタ
3. 各船宿の catch_page を取得（0.8秒待機）
   - タイムアウト: 10秒/リクエスト
   - 失敗時: ログ出力して skip
4. parser_type に応じたパーサーを呼び出し
5. catches_raw.json と同一フォーマット（15フィールド）のレコードを生成
6. catches_raw_direct.json に保存（重複排除: ship+date+trip_no+fish_raw）
```

---

## catches_raw_direct.json の保存ルール

1. **同一フォーマット**: catches_raw.json と同じ15フィールド
2. **別ファイル**: catches_raw.json には触らない。catches_raw_direct.json に独立保存
3. **重複排除キー**: `(ship, date, trip_no, fish_raw)` の組み合わせ（ファイル内での重複排除）
4. **ソース識別**: CSV生成時にファイル名から `source` 列を付与（"fishing-v" / "direct"）

## crawl.yml への追加

```yaml
# 既存のステップ（変更なし）
- name: Crawl fishing-v.jp
  run: python3 crawler.py

# 追加のステップ
- name: Crawl ship websites directly
  run: python3 direct_crawler.py
  timeout-minutes: 10
  continue-on-error: true
```

- `continue-on-error: true` により、direct_crawler.py が失敗しても既存のデプロイは続行
- timeout-minutes: 10 でハングアップを防止

---

## パーサー実装の原則

1. **標準ライブラリのみ**: crawler.py と同様、外部依存なし（html.parser / urllib）
2. **べき等性**: 同日2回実行しても結果が変わらない
3. **テスト可能**: 各パーサーはHTMLファイルを入力、dictリストを出力する純関数
4. **エラーハンドリング**: パース失敗は空リストを返す（全体を止めない）
