# 05_実装方針 — パーサー設計・統合方針

> **変更ガバナンス**: catches_raw.json / data/*.csv のフィールド変更はオーナー最終承認 + 関連MD責任者了解が必須。

---

## 基本方針

- **新スクリプト**: `direct_crawler.py`（crawler.py とは完全分離）
- **既存パイプラインを壊さない**: catches_raw.json の既存フォーマットに合わせてマージ
- **個別失敗で全体を止めない**: 船宿ごとに try/except、失敗は skip して続行

---

## パーサータイプ（更新版）

調査の結果、忠彦丸・一之瀬丸・米元釣船店（金沢八景）は全て **gyo.ne.jp プラットフォーム**を使用しており、同一構造であることが判明。

| タイプ | 対象 | 特徴 |
|--------|------|------|
| **type_gyo** | gyo.ne.jp 登録38船宿（忠彦丸・一之瀬丸・米元釣船店等） | 釣り物別セクション・7日制限・CID切替で全船宿対応 |
| type_a | 敷嶋丸等（シンプル表型） | — |
| type_b | 山本釣船店（リッチ列型） | — |
| type_c | たいぞう丸（ナラティブ型） | — |

**type_gyo 1本で gyo.ne.jp の38船宿すべてに対応できる。**
CID（船宿ID）を変えるだけでパーサーを再利用可能。

## type_gyo パーサーの仕様

### URL生成

```python
# 最新ページ（毎日クロール用）
URL_LATEST = "https://www.gyo.ne.jp/rep_tsuri_view%7CCID-{cid}.htm"

# 履歴ページ（過去6日分まで有効）
URL_HISTORY = "https://www.gyo.ne.jp/rep_tsuri_history_view%7CCID-{cid}%7Chdt-{hdt}%7Cdt-{dt}.htm"
# hdt = 取得したい日付 (YYYY/MM/DD)
# dt  = アクセス当日の日付 (YYYY/MM/DD) ← 必須。7日以上前はリダイレクトされる
```

### 釣り物セクション識別ロジック

全 `<table>` がフラットに並ぶため、セル内テキストでセクション種別を判定:

```
「乗合船」「仕立船」「限定」を含む → 船種ヘッダー（次セクション開始）
「出船時間」「納竿時間」を含む    → タックルテーブル（スキップ）
「日釣果」または「月日釣果」を含む → 釣果テーブル（パース対象）
「コメント」を含む               → コメントテーブル（kanso_raw）
```

### 釣果日付の処理

```
"４月６日釣果"       → 月・日を抽出、年はURL日付から補完
"6日釣果"           → 日のみ、月・年はURL日付から補完
"月日釣果"          → スキップ（出船なし / 未更新）
"4月tadahiko1日釣果" → typo除去後にパース
```

⚠️ **釣果日付 ≠ ページ日付の場合がある**（更新されない釣り物は古い日付のまま）。
釣果行の日付を優先し、ページURLの日付で上書きしない。

### catches_raw フィールドへのマッピング

| gyo.ne.jp 情報 | フィールド | 処理 |
|---------------|-----------|------|
| 釣果行の日付 | `date` | 年補完 |
| 釣り物名 | `fish_raw` | そのまま |
| 釣果テキスト（匹数） | `count_raw` | 正規表現抽出 |
| 釣果テキスト（サイズ） | `size_raw` | 正規表現抽出 |
| 午前/午後/複数隻情報 | `kanso_raw` | 付記 |
| 船長コメント | `kanso_raw` | 長文そのまま |

## ships.json への追加フィールド

```json
{
  "sid": null,
  "name": "忠彦丸",
  "area": "金沢漁港",
  "catch_page": "https://www.gyo.ne.jp/rep_tsuri_view%7CCID-tadahiko.htm",
  "parser_type": "type_gyo",
  "gyo_cid": "tadahiko"
}
```

| フィールド | 型 | 説明 |
|-----------|-----|------|
| catch_page | string / null | 釣果ページURL |
| parser_type | string / null | "type_gyo" / "type_a" / "type_b" / "type_c" / null |
| gyo_cid | string / null | gyo.ne.jp の CID（type_gyo のみ使用） |

- `parser_type` が null の船宿はスキップ（釣りビジョン経由のみ）

---

## direct_crawler.py の処理フロー

```
1. ships.json を読み込む
2. parser_type が設定されている船宿をフィルタ
3. 各船宿の catch_page を取得（0.8秒待機）
   - タイムアウト: 10秒/リクエスト
   - 失敗時: ログ出力して skip
4. parser_type に応じたパーサーを呼び出し
5. catches_raw.json と同一フォーマット（15フィールド）のレコードを生成
6. catches_raw_direct.json に保存（重複排除: ship+date+fish_raw）
```

---

## catches_raw_direct.json の保存ルール

1. **同一フォーマット**: catches_raw.json と同じ15フィールド
2. **別ファイル**: catches_raw.json には触らない。catches_raw_direct.json に独立保存
3. **重複排除キー**: `(ship, date, fish_raw)` の組み合わせ（trip_noは型によって意味が異なるため除外）
4. **ソース識別**: CSV生成時にファイル名から `source` 列を付与（"fishing-v" / "direct"）

---

## crawl.yml への追加

```yaml
- name: Crawl fishing-v.jp
  run: python3 crawler.py

- name: Crawl ship websites directly
  run: python3 direct_crawler.py
  timeout-minutes: 10
  continue-on-error: true
```

`continue-on-error: true` により、direct_crawler.py が失敗しても既存のデプロイは続行。

---

## パーサー実装の原則

1. **標準ライブラリのみ**: crawler.py と同様、外部依存なし（html.parser / urllib）
2. **べき等性**: 同日2回実行しても結果が変わらない
3. **テスト可能**: 各パーサーはHTMLを入力、dictリストを出力する純関数
4. **エラーハンドリング**: パース失敗は空リストを返す（全体を止めない）

## 開発順序（更新版）

| Phase | 対象 | パーサー | 変更点 |
|-------|------|---------|--------|
| 1 | gyo.ne.jp 汎用実装（忠彦丸で検証） | type_gyo.py | 旧「type_e 半日」→「type_gyo 1〜2日」 |
| 2 | 一之瀬丸・米元釣船店 | type_gyo.py（CID変更のみ） | gyo.ne.jp登録済みのため追加工数ほぼゼロ |
| 3 | 荒川屋・太田屋・弁天屋 | HP調査後に決定（gyo.ne.jp未登録の可能性） | — |
| 4 | 山本釣船店 | type_b.py | — |
| 5 | その他 gyo.ne.jp 38船宿 | type_gyo.py（CID変更のみ） | — |

**Phase 1 の工数が上がった分、Phase 2 はほぼゼロになる。**
