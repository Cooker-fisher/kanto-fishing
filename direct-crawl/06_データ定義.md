## 設計原則: raw json は生データの器、CSV が料理する

**catches_raw.json の15フィールドは変更しない。**

raw json は「HPから取れた生テキストをそのまま格納する器」であり、構造化・正規化は CSV 層の役割。

### 自社HPデータの既存フィールドへの収め方

| HP上の情報 | 収め先 | 理由 |
|-----------|-------|------|
| 水温（構造化フィールド） | `suion_raw` | 既存フィールド（現在全件null → 値が入る） |
| 水色（構造化フィールド） | `suishoku_raw` | 同上 |
| 天気・潮汐・タックル等（ナラティブ中） | `kanso_raw` | フリーテキストとして格納 → CSVで抽出 |
| 天気・潮汐等（構造化フィールド） | `tokki_raw` | 特記事項として格納 → CSVで抽出 |
| 匹数 | `count_raw` | 既存フィールド |
| サイズ | `size_raw` | 既存フィールド |
| ポイント名 | `point_raw` | 既存フィールド |

**null → 値が入る変更は影響低**（下流で null チェック済み）。

### CSV列の追加（raw json 変更なし）

CSV生成時に追加できる列の候補:

| CSV列 | 内容 | raw json変更 |
|-------|------|------------|
| source | "fishing-v" / "direct" | 不要（判定ロジックで付与） |
| weather | 天気 | 不要（kanso_raw/tokki_rawから抽出） |
| tide_type_reported | 船長報告の潮汐 | 不要（同上） |
| tackle | タックル情報 | 不要（同上） |

CSV列の追加は既存列に影響しないため、ガバナンス不要。

## boat_count の扱い（検討中）

### 問題

忠彦丸の「午前ライトアジ船(2隻高低) 8～61匹」は **2隻分の合算**。

### 方針

- `count_raw` に「8～61匹」をそのまま格納（raw jsonの原則通り）
- 「(2隻高低)」は `tokki_raw` に入れる
- CSV生成時に `boat_count` 列を付与（tokki_rawから抽出、デフォルト1）
- 分析時は `boat_count` で正規化

**→ raw json のフィールド追加は不要。CSV層で対応。**

## ソース識別（確定）

**ファイル名で区別する。**

| ファイル | ソース | 生成スクリプト |
|---------|-------|------------|
| `catches_raw.json` | 釣りビジョン経由 | crawler.py |
| `catches_raw_direct.json` | 自社HP直接 | direct_crawler.py |

- フォーマット（15フィールド）は同一
- CSV生成時に、どちらのファイルから読んだかで `source` 列を付与（"fishing-v" / "direct"）
- raw json 自体にsourceフィールドは追加しない

## 変更ガバナンス手順

**raw json のフィールド変更時のみ**以下のプロセスを適用:

1. 本ファイル（06_データ定義.md）に候補を追記
2. 07_パイプライン影響分析.md で影響範囲を評価
3. PIPELINE.md の変更インパクトマトリクスを更新
4. **オーナーの最終承認を取得**
5. **関連MD責任者（PIPELINE.md 管理者、redesign/ 責任者）の了解を取得**
6. 実装・テスト・デプロイ

**CSV列の追加は既存データに影響しないため、このプロセスは不要。**
